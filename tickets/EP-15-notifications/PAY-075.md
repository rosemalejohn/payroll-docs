# [GW] Payroll - Payroll Workflow Notifications

## Description

Implement specific notification classes for each step in the payroll workflow. When a payroll run transitions between states, the appropriate users are notified via the in-app notification system (PAY-074). This ensures that finance users know when a run needs review, owners know when a run is approved, payroll officers know when a run is finalized or returned, and employees know when their payslips are available.

## Technical Implementation

### Notifications
- **`app/Payroll/Notifications/PayrollRunSubmittedNotification.php`**
  - Triggered when: PayrollRun transitions from `draft` to `submitted`
  - Sent to: all users with `finance` role
  - Channel: database
  - Data payload: `{ title: "Payroll Run Submitted", message: "Payroll run '{run_name}' for period {period_start} – {period_end} has been submitted for review.", action_url: "/payroll-runs/{id}" }`

- **`app/Payroll/Notifications/PayrollRunApprovedNotification.php`**
  - Triggered when: finance user approves a run (under_review, approved)
  - Sent to: all users with `owner` role
  - Channel: database
  - Data payload: `{ title: "Payroll Run Approved", message: "Payroll run '{run_name}' for period {period_start} – {period_end} has been approved by {approver_name} and is ready for finalization.", action_url: "/payroll-runs/{id}" }`

- **`app/Payroll/Notifications/PayrollRunFinalizedNotification.php`**
  - Triggered when: owner finalizes a run
  - Sent to: the `payroll_officer` who originally submitted the run
  - Channel: database
  - Data payload: `{ title: "Payroll Run Finalized", message: "Payroll run '{run_name}' for period {period_start} – {period_end} has been finalized by {finalizer_name}.", action_url: "/payroll-runs/{id}" }`

- **`app/Payroll/Notifications/PayslipAvailableNotification.php`**
  - Triggered when: payroll run is finalized
  - Sent to: each employee who has a payslip in the finalized run
  - Channel: database
  - Data payload: `{ title: "Payslip Available", message: "Your payslip for period {period_start} – {period_end} is now available.", action_url: "/my-payslips/{payslip_id}" }`

- **`app/Payroll/Notifications/PayrollRunReturnedToDraftNotification.php`**
  - Triggered when: a run is returned to a previous state (e.g., under_review → draft or submitted → draft)
  - Sent to: `finance` users (if returned from under_review) or `payroll_officer` who submitted (if returned from submitted)
  - Channel: database
  - Data payload: `{ title: "Payroll Run Returned", message: "Payroll run '{run_name}' for period {period_start} – {period_end} has been returned to {new_state} by {actor_name}. Reason: {reason}.", action_url: "/payroll-runs/{id}" }`

### Events & Listeners
- **`app/Payroll/Events/PayrollRunSubmittedEvent.php`** → **`app/Payroll/Listeners/SendPayrollRunSubmittedNotificationListener.php`**
  - Queries all users with `finance` role
  - Sends `PayrollRunSubmittedNotification` to each

- **`app/Payroll/Events/PayrollRunApprovedEvent.php`** → **`app/Payroll/Listeners/SendPayrollRunApprovedNotificationListener.php`**
  - Queries all users with `owner` role
  - Sends `PayrollRunApprovedNotification` to each

- **`app/Payroll/Events/PayrollRunFinalizedEvent.php`** → **`app/Payroll/Listeners/SendPayrollRunFinalizedNotificationListener.php`**
  - Identifies the payroll_officer who submitted the run (from `submitted_by` field)
  - Sends `PayrollRunFinalizedNotification` to that user

- **`app/Payroll/Events/PayrollRunFinalizedEvent.php`** → **`app/Payroll/Listeners/SendPayslipAvailableNotificationListener.php`**
  - Queries all employees with payslips in the finalized run
  - Sends `PayslipAvailableNotification` to each employee user account
  - Should be dispatched to a queue to avoid blocking finalization for large runs

- **`app/Payroll/Events/PayrollRunReturnedEvent.php`** → **`app/Payroll/Listeners/SendPayrollRunReturnedNotificationListener.php`**
  - Determines recipients based on the previous state
  - Sends `PayrollRunReturnedToDraftNotification` to appropriate users

### Jobs
- **`app/Payroll/Jobs/SendBulkPayslipNotificationsJob.php`** — Queued job for sending payslip notifications to all employees in a finalized run:
  - Accepts payroll run ID
  - Iterates through all payslips in the run
  - Sends `PayslipAvailableNotification` to each employee
  - Queue: `notifications`

### Tests
- **`tests/Feature/Payroll/PayrollWorkflowNotificationTest.php`**
  - Submitting a run sends notification to all finance users
  - Approving a run sends notification to all owner users
  - Finalizing a run sends notification to the submitting payroll_officer
  - Finalizing a run sends payslip notifications to all employees in the run
  - Returning a run to draft sends notification to appropriate users based on previous state
  - Notification data includes run name, period, and action URL
  - Notifications appear in the recipient's notification bell dropdown
  - Users who are not recipients do not receive notifications
  - Payslip notifications are dispatched to the queue (not synchronous)

### Edge Cases & Notes
- For large payroll runs (hundreds of employees), payslip notifications must be queued to avoid timeouts during finalization
- If a user has multiple roles (e.g., owner + finance), they should not receive duplicate notifications for the same event
- The `PayrollRunReturnedToDraftNotification` must handle multiple possible previous states and route to the correct recipients
- If an employee does not have a user account (no login access), their payslip notification is skipped silently
- Notification content must include enough context (run name, period) so the recipient understands the notification without clicking through

## Acceptance Criteria

- [ ] Finance users receive notification when a payroll run is submitted
- [ ] Owner users receive notification when a payroll run is approved by finance
- [ ] Payroll officer who submitted the run receives notification when it is finalized
- [ ] Each employee with a payslip in a finalized run receives a "Payslip Available" notification
- [ ] Appropriate users receive notification when a run is returned to draft
- [ ] Notification content includes the run name and period dates
- [ ] Notifications include an action URL that links to the relevant page
- [ ] Notifications appear in the bell icon dropdown for recipients
- [ ] Payslip notifications are dispatched to a queue for performance
- [ ] Users with multiple roles do not receive duplicate notifications
- [ ] All workflow notification tests pass
