# PAY-056: Cash Advance Recovery Post-Finalization

## Description

Implement the background job that updates cash advance balances after a payroll run is finalized. When a payroll run transitions to `finalized`, a job processes all payslips that have a `cash_advance_deduction > 0`. For each such payslip, the job finds the employee's active cash advances (ordered by `issued_date` ascending for FIFO recovery) and applies the deducted amount, updating `amount_recovered` and `balance`. When a cash advance's balance reaches zero, its status is automatically set to `fully_recovered`.

This job ensures that payroll deductions are reflected back in the cash advance tracking system after finalization.

## Technical Implementation

### Job

**`app/CashAdvance/Jobs/RecoverCashAdvancesJob.php`**

- Dispatched by `FinalizePayrollRunAction` after the run state transitions to `finalized`.
- Queue: `default` (or a dedicated `payroll` queue).
- Accepts: `PayrollRun $payrollRun`

**Logic:**

```php
public function handle(): void
{
    $payslips = Payslip::where('payroll_run_id', $this->payrollRun->id)
        ->where('cash_advance_deduction', '>', 0)
        ->get();

    foreach ($payslips as $payslip) {
        $remainingDeduction = $payslip->cash_advance_deduction;

        // FIFO: oldest issued cash advances first
        $activeAdvances = CashAdvance::where('employee_id', $payslip->employee_id)
            ->where('status', 'active')
            ->where('balance', '>', 0)
            ->orderBy('issued_date', 'asc')
            ->orderBy('created_at', 'asc')
            ->get();

        foreach ($activeAdvances as $advance) {
            if ($remainingDeduction <= 0) {
                break;
            }

            $recoveryAmount = min($remainingDeduction, $advance->balance);

            $advance->amount_recovered += $recoveryAmount;
            $advance->balance -= $recoveryAmount;

            if ($advance->balance <= 0) {
                $advance->balance = 0;
                $advance->status = 'fully_recovered';
            }

            $advance->save();

            $remainingDeduction -= $recoveryAmount;
        }
    }
}
```

### Integration Point

In `FinalizePayrollRunAction`, after the state transition and locking:

```php
// After finalization
RecoverCashAdvancesJob::dispatch($payrollRun);
```

### Edge Cases and Business Rules

- **FIFO order**: Cash advances are recovered in order of `issued_date` ascending, then `created_at` ascending as a tiebreaker. The oldest advance is paid off first.
- **Partial recovery**: If the deduction amount is less than the advance balance, only a partial recovery is applied. The remaining balance carries forward to the next payroll run.
- **Multiple advances**: If an employee has multiple active cash advances and the deduction covers more than one, recovery cascades through them in FIFO order.
- **Over-recovery safety**: The `min($remainingDeduction, $advance->balance)` guard prevents recovering more than the advance balance.
- **Zero balance auto-status**: When `balance` reaches 0, status is automatically set to `fully_recovered`.
- **Idempotency**: The job should be idempotent. Running it twice for the same finalized run should not double-recover. This can be enforced by checking if recovery was already processed (e.g., via a `recovery_processed_at` timestamp on the payroll run or by verifying the audit trail).
- **Cancelled advances**: Only `active` advances are considered for recovery. Cancelled advances are skipped.
- **Database transactions**: Each employee's recovery should be wrapped in a transaction to ensure consistency.

### Audit Trail

Each `CashAdvance` model implements `Auditable`, so every update to `amount_recovered`, `balance`, and `status` is automatically logged with the old and new values. The audit records link recovery events back to the payroll run through the job context.

### Tests

- `tests/Feature/CashAdvance/RecoverCashAdvancesJobTest.php`
  - Single advance fully recovered: balance becomes 0, status changes to `fully_recovered`.
  - Single advance partially recovered: balance reduced, status remains `active`.
  - Multiple advances FIFO: oldest advance recovered first, then next.
  - Multiple advances with full cascade: deduction covers first advance fully and partially covers second.
  - Employee with no active advances: job completes without error.
  - Payslip with zero cash_advance_deduction: skipped.
  - Cancelled advance is not considered for recovery.
  - Job is idempotent: running twice does not double-recover.
  - Balance never goes negative (floored at 0).
  - Audit trail records recovery amounts and status changes.

## Acceptance Criteria

- [ ] After payroll run finalization, cash advance balances are updated for all employees with `cash_advance_deduction > 0`.
- [ ] Recovery follows FIFO order based on `issued_date` ascending (oldest advance recovered first).
- [ ] `amount_recovered` is incremented and `balance` is decremented by the recovery amount.
- [ ] Cash advances with balance reaching zero are automatically set to `status = 'fully_recovered'`.
- [ ] Partial recovery is handled correctly — remaining balance carries to the next payroll cycle.
- [ ] Recovery cascades across multiple active advances for the same employee in FIFO order.
- [ ] Cancelled cash advances are not considered for recovery.
- [ ] The job is dispatched automatically after `FinalizePayrollRunAction` completes.
- [ ] The job is idempotent and does not double-recover if executed multiple times.
- [ ] All recovery changes are captured in the audit trail via `owen-it/laravel-auditing`.
