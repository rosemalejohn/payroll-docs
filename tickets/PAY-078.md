# PAY-078: Currency Rate Update Mid-Run Warning

## Description

When an owner updates a currency exchange rate, the system must detect if any open payroll runs (draft, submitted, or under_review) contain employees paid in that currency. If affected runs exist, the owner is presented with a warning dialog listing the impacted runs and given two options: (1) apply the new rate to the current run by re-snapshotting and recomputing affected payslips, or (2) keep the existing snapshot rate for current runs and apply the new rate only to future runs. All decisions are logged in the audit trail for accountability.

## Technical Implementation

### Actions
- **`app/Currency/Actions/UpdateCurrencyRateAction.php`** — Enhanced to detect affected runs:
  - Accepts: currency ID, new rate
  - Before updating, calls `FindAffectedPayrollRunsQuery` to check for open runs with employees in that currency
  - If affected runs found: returns the list of affected runs to the controller (does not auto-apply)
  - If no affected runs: updates the rate directly

- **`app/Currency/Actions/ApplyRateToOpenRunAction.php`** — Applies the new rate to a specific open run:
  - Updates the `payroll_run_currency_snapshots` record for this run and currency
  - Calls `RecomputeAffectedPayslipsAction` for all payslips in the run using this currency
  - Sets a flag/note on the run: "Currency rate for {code} updated on {date}. Figures recomputed."
  - Creates audit entry with old rate, new rate, run ID, and user decision

- **`app/Payroll/Actions/RecomputeAffectedPayslipsAction.php`** — Recomputes payslips after rate change:
  - Accepts: payroll run ID, currency ID
  - Finds all payslips for employees whose `salary_currency_id` matches
  - Runs `ComputePayslipAction` for each affected payslip with the updated snapshot rate
  - Returns count of recomputed payslips

### Queries
- **`app/Currency/Queries/FindAffectedPayrollRunsQuery.php`** — Finds open runs affected by a currency change:
  - Queries payroll runs in `draft`, `submitted`, or `under_review` state
  - Joins through payslips → employees to find runs containing employees with the given `salary_currency_id`
  - Returns collection with run `id`, `name`, `status`, `period_start`, `period_end`, and count of affected employees

### Validation & DTOs
- **`app/Currency/DTOs/CurrencyRateUpdateDecisionData.php`** — DTO for the decision:
  - `currency_id`: string (ULID)
  - `new_rate`: decimal
  - `apply_to_run_ids`: array of ULID strings (runs to apply new rate to)
  - `decision`: enum ('apply_to_current', 'future_only')

- **`app/Http/Requests/Currency/UpdateCurrencyRateRequest.php`** — Validates:
  - `rate_to_php`: required, numeric, gt:0
  - `decision`: required_if affected runs exist, in:apply_to_current,future_only
  - `apply_to_run_ids`: required_if decision is apply_to_current, array of existing payroll run ULIDs

### Routes & Controllers
- **`PUT /currencies/{currency}/rate`** — Route name: `currencies.rate.update`
- **`POST /currencies/{currency}/rate/apply-to-runs`** — Route name: `currencies.rate.apply-to-runs`
- **`app/Http/Controllers/Web/CurrencyRateController`**
  - `update()`: checks for affected runs; if found, returns Inertia response with affected runs data for the warning dialog; if none, updates rate and redirects
  - `applyToRuns()`: applies rate to selected open runs, recomputes payslips, redirects with flash message
  - Middleware: `auth`, `role:owner`

### Frontend
- **`resources/js/components/currency/rate-change-warning-dialog.tsx`** — Warning dialog:
  - Displays list of affected payroll runs with name, status, period, and employee count
  - Option 1 button: "Apply to Current Runs" — applies new rate and triggers recomputation
  - Option 2 button: "Future Runs Only" — keeps current snapshot, applies rate for future runs
  - Each option explains the consequence clearly
  - Confirm step before applying (especially Option 1, which changes computed figures)

- **`resources/js/pages/payroll-runs/show.tsx`** — Add rate change banner:
  - If the run has a `currency_rate_updated_at` flag, show an info banner: "Currency rate for {code} was updated on {date}. Figures have been recomputed."
  - Banner is dismissible but persists across page loads (tied to run data, not session)

### Models
- **`app/Payroll/Models/PayrollRun.php`** — Add:
  - `currency_rate_notes` (nullable text) to `$fillable` — stores rate change log

### Migrations
- **`database/migrations/xxxx_add_currency_rate_notes_to_payroll_runs_table.php`**
  - Add `currency_rate_notes` (text, nullable)

### Tests
- **`tests/Feature/Currency/CurrencyRateMidRunWarningTest.php`**
  - Updating rate with no open runs proceeds without warning
  - Updating rate with affected open runs returns warning with run list
  - Choosing "apply to current" re-snapshots and recomputes affected payslips
  - Choosing "future only" does not change existing snapshots
  - Recomputed payslips have correct values based on new rate
  - Banner appears on run detail page after rate is applied
  - Audit trail captures old rate, new rate, affected runs, and user decision
  - Only owner role can update currency rates
  - Non-owner roles receive 403
  - Multiple affected runs can be selected for rate application

### Edge Cases & Notes
- If a run is in `under_review` state and the rate is applied, finance should be alerted that figures have changed (consider sending a notification)
- The recomputation must update the snapshot table first, then recompute payslips — ordering matters
- If recomputation fails for any payslip (e.g., missing data), the entire operation should be wrapped in a database transaction and rolled back
- Currency rate changes for PHP (Philippine Peso) should be blocked since it is the immutable base currency (rate always 1.0)
- The audit record must capture the user's explicit decision (apply or future-only) for compliance purposes

## Acceptance Criteria

- [ ] Updating a currency rate with affected open runs shows a warning dialog listing impacted runs
- [ ] Warning shows run name, status, period, and count of affected employees per run
- [ ] Option 1 ("Apply to Current Runs") re-snapshots the rate and recomputes all affected payslips
- [ ] Option 2 ("Future Runs Only") preserves existing snapshot rates and only applies to new runs
- [ ] Recomputed payslips reflect the new currency rate correctly
- [ ] Info banner displayed on affected run detail page after rate application
- [ ] Audit trail captures: old rate, new rate, affected run(s), user decision
- [ ] Only owner role can update currency rates
- [ ] PHP (Philippine Peso) rate cannot be changed
- [ ] Recomputation is wrapped in a transaction for atomicity
- [ ] All currency rate mid-run tests pass
