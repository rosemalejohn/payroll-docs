# [GW] Payroll - Employee Deactivation with Confirmation

## Description

Build the employee deactivation flow with a confirmation step that warns about open payroll runs. Deactivated employees are excluded from future payroll runs while their historical data is preserved. No data is deleted.

## Technical Implementation

### What Actions

- Deactivate Employee action: sets the employee status to inactive. Before deactivating, checks for open payroll runs (draft, submitted, or under review) that include this employee and returns the list for the confirmation UI.

### Routes & Controllers

- `GET /employees/{employee}/check-deactivation` — returns a list of affected open payroll runs for the confirmation dialog.
- `PATCH /employees/{employee}/deactivate` — performs the deactivation.

### Policies & Authorization

- Owner and Payroll Officer can deactivate employees. Other roles are denied.

### Tests

- Assert employee status changes to inactive after deactivation.
- Assert confirmation check returns affected open payroll runs when they exist.
- Assert deactivated employees are excluded from new payroll run auto-population.
- Assert historical payslips remain accessible and unchanged after deactivation.
- Assert employee data is not deleted (soft deactivation only).
- Assert Owner and Payroll Officer can deactivate; other roles receive 403.

## Acceptance Criteria

### Client-side

#### Behavior

- A "Deactivate" button is available on the employee profile for Owner and Payroll Officer roles
- Before deactivating, the system checks for open payroll runs containing this employee
- If open payroll runs are found, a confirmation dialog lists the affected runs with a warning
- If no open payroll runs are found, a simple "Are you sure?" confirmation is shown
- After confirming, the employee's status changes to inactive

---

### Server-side

#### Side-effects

- Deactivating an employee sets their status to inactive — no data is deleted
- Deactivated employees are automatically excluded from future payroll run auto-population
- Existing payslips in open payroll runs are NOT removed — manual removal is still required if needed
- All historical payslips of deactivated employees remain accessible for viewing and reporting
