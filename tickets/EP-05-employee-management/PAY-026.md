# [GW] Payroll - Employee List Page with Filters and Pagination

## Description

Build the employee list page — the primary view for browsing and managing employees. This includes creating the employees table and model (the local mirror of HRIS data with payroll-specific fields), along with the list page controller, query class, form request, API resource, and Inertia React page using the reusable data table component. Access is restricted to Owner and Payroll Officer roles.

## Technical Implementation

### What Migrations

- Create the employees table with ULID primary key, company reference, HRIS employee ID (unique — used as sync key), personal info fields (employee number, first name, last name, middle name, email), employment details (department, position, employment type, employment status, date hired), payroll-specific local-only fields (pay basis, external salary flag, MP2 flag, SSS WISP flag, HMO amount, government ID numbers), and last synced timestamp.

### What Models

- Create Employee model in the Employee domain with ULID support, factory, and audit trail integration.
- The model separates HRIS-synced fields (overwritten on sync) from payroll-specific fields (managed locally, never overwritten).
- Audit trail is configured to track only payroll-specific field changes to avoid noise from frequent syncs.
- Scopes: active employees, filter by department, filter by company.
- Accessor for full name formatting.
- Factory includes state methods for daily pay basis, external salary, inactive, and terminated.

### Queries

- Dedicated query class for building the filtered, sorted, and paginated employee list. Supports search (by name or employee number), filters for department, employment status, and pay basis, and sorting by common columns with default sort by last name.

### Validation & DTOs

- Form request validating query string parameters: search, department, employment status, pay basis, sort column, sort direction, per page, and page number.

### Routes & Controllers

- `GET /employees` — paginated employee list with filters.
- Controller returns Inertia page with paginated employees (via API resource), current filter values, and distinct department values for the dropdown.

### API Resources

- Employee list resource shaping employee data for the list page: ID, employee number, name, email, department, position, pay basis, employment status, employment type, external salary flag, and date hired.

### Policies & Authorization

- Route-level middleware restricts access to Owner and Payroll Officer roles. Employee and Finance roles receive 403.

### Tests

- Assert Owner and Payroll Officer can access the employee list.
- Assert Employee and Finance roles receive 403.
- Assert unauthenticated users are redirected to login.
- Assert search filter returns matching employees by name and employee number.
- Assert department, status, and pay basis filters narrow results correctly.
- Assert pagination and sorting work correctly.
- Assert empty state is returned when no employees match filters.
- Assert employee model can be created with valid attributes and ULID primary key.
- Assert HRIS employee ID has a unique constraint.
- Assert audit trail tracks payroll-specific field changes but not HRIS-synced field changes.
- Assert factory and state methods produce valid data.

### Edge Cases & Notes

- Search should be case-insensitive.
- Department dropdown values should be dynamically derived from existing employee data, not hardcoded.
- Deactivated employees should still appear in the list (filtered by status).

## Acceptance Criteria

### Client-side

#### Layout

- The Employees page is accessible from the sidebar navigation for Owner and Payroll Officer roles

#### Behavior

- The employee list displays in a table with columns: Employee Number, Name, Department, Position, Pay Basis, Status, and an External Salary indicator
- Users can search employees by name or employee number using the search bar
- Users can filter the list by department, employment status, and pay basis using dropdown filters
- Clicking a sortable column header sorts the table by that column
- Pagination controls allow navigating between pages and changing the number of results per page
- All filter, sort, and pagination settings are reflected in the URL and can be bookmarked
- An "Add Employee" button links to the employee creation page
- An empty state message appears when no employees exist or no results match the current filters

---

### Server-side

#### Side-effects

- Only Owner and Payroll Officer roles can access the employee list — Employee and Finance roles are denied
- Employee data is stored locally with both HRIS-synced fields and payroll-specific fields
- Changes to payroll-specific fields (external salary, MP2, SSS WISP, HMO) are tracked in the audit trail
- Changes to HRIS-synced fields (name, department, position) are not tracked in the audit trail to avoid noise from daily syncs
