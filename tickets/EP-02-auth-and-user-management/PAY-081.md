# [GW] Payroll - Payroll Officer Finalization Access Restriction

## Description

Enforce that users with the `payroll_officer` role cannot finalize payroll runs. The restriction must be implemented at the policy layer (`PayrollRunPolicy@finalize`) so it cannot be bypassed by direct API calls. The finalize button must be hidden in the UI for payroll officers, and any attempt to call the finalize endpoint must return a 403 Forbidden response.

Payroll officers should still be able to see the current status of a payroll run, including when it is pending finalization. When a run they submitted is finalized by the owner, they receive an in-app notification confirming the finalization.

## Technical Implementation

### Actions
- No new actions required — this ticket hardens the existing `FinalizePayrollRunAction` by ensuring the policy layer blocks payroll officers.

### Policies & Authorization
- **`app/Payroll/Policies/PayrollRunPolicy.php`** — Update `finalize()` method:
  - Explicitly returns `false` when `$user->hasRole('payroll_officer')`
  - Only `owner` role returns `true` (already implemented in PAY-062, this ticket verifies and hardens)
  - Policy is registered in `AuthServiceProvider` and applied via controller authorization

### Routes & Controllers
- **`POST /payroll-runs/{payrollRun}/finalize`** — Existing route (PAY-062)
- **`app/Http/Controllers/Web/PayrollRunFinalizationController@store`**
  - Ensure `$this->authorize('finalize', $payrollRun)` is called before any action execution
  - Middleware: `auth`, `role:owner` provides first-layer protection; policy provides second-layer

### Frontend (Inertia/React)
- **`resources/js/pages/payroll-runs/show.tsx`** — Finalize button rendering:
  - Pass `can.finalize` boolean from controller via Inertia page props
  - Conditionally render the finalize button: `{can.finalize && <FinalizeButton />}`
  - Payroll officer sees the run status badge (e.g., "Under Review — Pending Finalization") but no finalize action
- **`resources/js/pages/payroll-runs/index.tsx`** — Status column shows current state for all roles; no finalize action link for payroll officers

### Events & Listeners
- **`app/Payroll/Listeners/NotifyStakeholdersOfFinalizationListener.php`** — Ensure the existing finalization listener (PAY-062) includes notification to the payroll officer who originally submitted the run:
  - Look up the `submitted_by` field on the payroll run
  - Send `PayrollRunFinalizedForOfficerNotification` to that user

### Notifications
- **`app/Payroll/Notifications/PayrollRunFinalizedForOfficerNotification.php`** — Database notification to the submitting payroll officer containing:
  - Payroll run name and period
  - Finalized by (owner name)
  - Finalized at timestamp
  - Link to view the finalized run

### Tests
- **`tests/Feature/Payroll/PayrollOfficerFinalizationRestrictionTest.php`**
  - Payroll officer cannot finalize a payroll run (403 response)
  - Finalize button is not present in Inertia props (`can.finalize = false`) for payroll officer
  - Payroll officer can view the run detail page and see the current status
  - Payroll officer receives notification when their submitted run is finalized
  - Owner can still finalize successfully (control test)
  - Finance role also cannot finalize (403 response)
  - Unauthenticated user is redirected to login

### Edge Cases & Notes
- Policy enforcement is the authoritative layer — even if middleware is misconfigured, the policy must block unauthorized finalization
- The `can.finalize` prop must be derived from the policy, not hardcoded role checks in the controller
- If a payroll officer somehow obtains the finalize URL and submits a POST request, the 403 must be returned before any state mutation occurs

## Acceptance Criteria

- [ ] Finalize button is not visible to users with the `payroll_officer` role
- [ ] Direct API call to `POST /payroll-runs/{id}/finalize` by payroll officer returns 403 Forbidden
- [ ] Payroll officer can view the payroll run detail page and see its current status
- [ ] Payroll officer receives an in-app notification when their submitted run is finalized
- [ ] `PayrollRunPolicy@finalize` explicitly returns `false` for `payroll_officer` role
- [ ] Policy enforcement is at the backend layer, not just UI hiding
- [ ] Owner can still finalize payroll runs successfully
