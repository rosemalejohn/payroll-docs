# [GW] Payroll - Currency Management Page (Settings)

## Description

Build the currency management settings page where the Owner can view and manage currencies, and create the currencies and currency snapshots database tables. The currencies table stores exchange rates as "1 [FOREIGN] = X PHP" with high decimal precision. PHP (Philippine Peso) is the immutable base currency. The currency snapshot table captures all active currency rates at the time a payroll run is created, ensuring historical integrity. Non-owner roles can view currencies in read-only mode.

## Technical Implementation

### What Migrations

- Create the currencies table with ULID primary key, company reference, code (ISO 4217, unique per company), name, symbol, rate (high-precision decimal for exchange rates), base currency flag, active flag, rate update timestamp, and standard timestamps.
- Create the payroll run currency snapshots table with ULID primary key, payroll run reference, currency reference, denormalized currency code, snapshot rate, and snapshot timestamp.

### What Models

- Create Currency model in the Currency domain with ULID support, factory, conversion helpers (to PHP and from PHP), and audit trail integration. Factory includes states for base currency and inactive.
- Create PayrollRunCurrencySnapshot model in the Payroll domain with ULID support and factory.

### Routes & Controllers

- `GET /settings/currencies` — paginated currency list with rate, last updated, and active status.
- Owner sees edit and toggle buttons; non-owner sees read-only table.

### Policies & Authorization

- All authenticated users can view currencies.
- Only the Owner role can create, update rates, or toggle active status.

### Tests

- Assert all authenticated users can view the currency list.
- Assert only Owner sees action buttons.
- Assert PHP base currency has no edit/toggle controls.
- Assert currency model uses ULID primary keys.
- Assert unique constraint on company and code prevents duplicates.
- Assert conversion helpers calculate correctly.
- Assert base currency rate is always 1.0 and immutable.
- Assert currency snapshots can be created and linked to payroll runs.

### Edge Cases & Notes

- Rates are stored with 6 decimal places for precision.
- PHP base currency cannot be edited, deactivated, or deleted.
- The rate format convention is "1 [FOREIGN] = X PHP".

## Acceptance Criteria

### Client-side

#### Layout

- The Currency Management page is accessible from the Settings section in the sidebar

#### Behavior

- The currency list displays each currency's code, name, symbol, exchange rate (with 6 decimal places), relative "last updated" timestamp, and active status
- The Owner sees Edit Rate and Toggle Active buttons on each currency row (except PHP)
- Non-owner roles see a read-only table with no action buttons
- The PHP (Philippine Peso) row is always present, always active, and has no edit or toggle controls
- Rates are displayed in the format "1 [CODE] = ₱[rate]"

---

### Server-side

#### Side-effects

- PHP (Philippine Peso) is the base currency with a fixed rate of 1.0 that cannot be changed
- Currency codes must be unique within the company — duplicates are rejected
- Currency rate changes are tracked in the audit trail with old and new values
- When a payroll run is created, all active currency rates are captured in a snapshot for historical integrity
- Snapshot rates are immutable after creation
