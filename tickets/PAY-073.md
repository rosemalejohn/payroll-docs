# PAY-073: Payroll Run Audit History Tab

## Description

Add an "Audit History" tab to the payroll run detail page that displays all audit entries related to a specific payroll run. This includes state transitions with actor and timestamp, bonus changes, employee additions/removals, and payslip recomputations. The tab presents entries in a timeline-style layout with the newest events first, giving payroll officers, finance users, and owners a clear chronological record of everything that happened to a run.

## Technical Implementation

### Actions
- **`app/Payroll/Actions/GetPayrollRunAuditHistoryAction.php`** — Retrieves all audit entries related to a specific payroll run:
  - Fetches audits where `auditable_type` is PayrollRun and `auditable_id` matches the run
  - Fetches audits where `auditable_type` is Payslip and the payslip belongs to the run
  - Fetches audits where `auditable_type` is PayrollRunBonus (or similar) linked to the run
  - Merges and sorts all results by `created_at` descending
  - Eager loads actor (user) relationship on each audit

### Queries
- **`app/Payroll/Queries/PayrollRunAuditQuery.php`** — Builds the union query across multiple auditable types:
  - Query 1: `audits` where `auditable_type = PayrollRun` and `auditable_id = {run_id}`
  - Query 2: `audits` where `auditable_type = Payslip` and `auditable_id IN (payslip IDs for this run)`
  - Query 3: `audits` where `auditable_type = PayrollRunBonus` and `auditable_id IN (bonus IDs for this run)`
  - Combines with `UNION ALL`, orders by `created_at DESC`
  - Returns paginated results (20 per page)

### Routes & Controllers
- **`GET /payroll-runs/{payrollRun}/audit-history`** — Route name: `payroll-runs.audit-history`
- **`app/Http/Controllers/Web/PayrollRunAuditHistoryController@index`**
  - Middleware: `auth`, `role:owner|payroll_officer|finance`
  - Authorizes via `PayrollRunPolicy@viewAuditHistory`
  - Delegates to `GetPayrollRunAuditHistoryAction`
  - Returns audit entries as Inertia partial reload data or full page props for the tab

### API Resources
- **`app/Http/Resources/PayrollRunAuditEntryResource.php`** — Formats audit entries for the timeline:
  - `id`, `event`, `auditable_type` (human-readable label)
  - `description` — generated summary (e.g., "Status changed from draft to submitted", "Payslip for John Doe recomputed", "Bonus 13th Month added")
  - `old_values`, `new_values` (relevant fields only)
  - `user` (actor name and role)
  - `created_at` (formatted timestamp with relative time, e.g., "2 hours ago")
  - `is_state_transition` (boolean flag for prominent styling)

### Policies & Authorization
- **`app/Payroll/Policies/PayrollRunPolicy.php`** — Add `viewAuditHistory()` method:
  - `owner`, `payroll_officer`, and `finance` roles can view
  - `employee` role cannot view run audit history

### Frontend
- **`resources/js/pages/payroll-runs/partials/audit-history-tab.tsx`** — Timeline-style audit history:
  - Each entry as a timeline node with timestamp, actor avatar/name, and event description
  - State transitions displayed prominently with colored badges (e.g., draft → submitted in blue, finalized in green)
  - Expandable detail section for each entry showing old/new values
  - "Load more" pagination or infinite scroll
  - Empty state: "No audit history available"

- **`resources/js/components/audit/audit-timeline-entry.tsx`** — Single timeline entry component:
  - Icon based on event type (state change, edit, create)
  - Actor name and role badge
  - Relative timestamp with full timestamp on hover
  - Collapsible detail panel for old/new values

- **`resources/js/types/payroll-run-audit.ts`** — TypeScript types for PayrollRunAuditEntry

### Tests
- **`tests/Feature/Payroll/PayrollRunAuditHistoryTest.php`**

  - Payroll officer can view audit history tab
  - Finance user can view audit history tab
  - Employee role receives 403
  - Unauthenticated user is redirected to login
  - State transitions appear in the audit history
  - Payslip changes related to the run appear in the audit history
  - Bonus additions/removals appear in the audit history
  - Entries are sorted newest first
  - Results are paginated
  - Audit entries include correct actor information

### Edge Cases & Notes
- The union query across multiple auditable types may need careful indexing; ensure `auditable_type` and `auditable_id` are indexed in the `audits` table
- State transition entries should be visually distinct from regular field changes to help users quickly scan the timeline
- If a payroll run has hundreds of payslip audits (e.g., large company), pagination is critical to prevent performance issues
- Consider caching the payslip IDs for the run to avoid repeated lookups when building the query
