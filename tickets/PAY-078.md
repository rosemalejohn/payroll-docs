# [GW] Payroll - Currency Rate Update Mid-Run Warning

## Description

When a currency rate is updated, the system must detect any open payroll runs (draft, submitted, under_review) containing employees paid in that currency. If affected runs exist, show a warning dialog listing those runs and let the owner choose:

1. **Apply to Current Runs** — update the snapshot rate for those runs and recompute affected payslips.  
2. **Future Runs Only** — keep existing snapshots and apply the new rate only going forward.

All decisions must be logged in the audit trail.

## Technical Implementation

### Actions
- **`UpdateCurrencyRateAction`**  
  - Detects open runs that use the currency via `FindAffectedPayrollRunsQuery`.  
  - If none: update rate immediately.  
  - If any: return list to controller for warning dialog.

- **`ApplyRateToOpenRunAction`**  
  - Updates the run’s currency snapshot.  
  - Calls `RecomputeAffectedPayslipsAction` for payslips using that currency.  
  - Records a note and audit entry with old rate, new rate, and decision.

- **`RecomputeAffectedPayslipsAction`**  
  - Recomputes only the payslips affected by the rate change and updates totals using adjustments.

### Queries
- **`FindAffectedPayrollRunsQuery`**  
  - Finds open runs that include employees with the given salary currency.  
  - Returns run info plus count of affected employees.

### Validation & DTOs
- **`CurrencyRateUpdateDecisionData`**  
  - Contains: `currency_id`, `new_rate`, `decision`, and selected `run_ids`.  
- **`UpdateCurrencyRateRequest`**  
  - Validates `rate_to_php`, user decision, and submitted run IDs.

### Routes & Controllers
- `PUT /currencies/{currency}/rate` — checks for affected runs; returns warning dialog if any.  
- `POST /currencies/{currency}/rate/apply-to-runs` — applies rate to chosen runs and recomputes.  
- Restricted to owners.

### Frontend
- **Warning Dialog**  
  - Lists affected runs with status, period, and employee count.  
  - Buttons: *Apply to Current Runs* or *Future Runs Only*.  
  - Confirmation step for the apply-to-current option.

- **Run Page Banner**  
  - Shows: “Currency rate for {code} was updated on {date}. Figures recomputed.”  

### Models / Migrations
- Add `currency_rate_notes` (nullable text) to `payroll_runs` to store audit notes.

### Tests
- No open runs → update proceeds directly.  
- With affected runs → warning shown.  
- Apply-to-current → snapshot updated + payslips recomputed.  
- Future-only → snapshots unchanged.  
- Banner appears after application.  
- Audit logs all decisions.  
- Only owners allowed; others receive 403.  
- PHP base currency cannot be changed.

### Edge Cases
- Operations must be wrapped in transactions.  
- Snapshot update must occur before recomputation.  
- Under-review runs may need finance notification (optional).

## Acceptance Criteria

- [ ] Warning dialog appears when updating a rate that affects open runs  
- [ ] Dialog lists impacted runs with status, period, and employee count  
- [ ] Apply-to-current resnapshots and recomputes affected payslips  
- [ ] Future-only leaves snapshots unchanged  
- [ ] Recomputed payslips use the new rate  
- [ ] Run page shows currency-rate-updated banner  
- [ ] Audit logs old rate, new rate, run IDs, and decision  
- [ ] Only owners can update rates  
- [ ] PHP currency remains locked  
- [ ] Transactional safety guaranteed  
- [ ] All related tests pass
