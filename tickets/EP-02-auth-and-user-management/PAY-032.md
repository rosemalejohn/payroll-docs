# [GW] Payroll - User Management Page (Owner Only)

## Description

Build the user management page where the owner can create users, assign roles, and toggle active status. Only the owner role can access this section. This provides full user account lifecycle management.

## Technical Implementation

### What Actions

- Create User action: creates a new user account, assigns the selected role, and hashes the password.
- Update User action: updates user information and role. Includes a guard to prevent removing the last owner account.

### Validation & DTOs

- Store User request: validates name (required), email (required, unique), password (required, min 8 characters), and role (required, must be one of the 4 defined roles).
- Update User request: validates name, email (unique except current user), role, and active status.

### Routes & Controllers

- `GET /users` — paginated user list with filters
- `GET /users/create` — create user form
- `POST /users` — store new user
- `GET /users/{user}/edit` — edit user form
- `PUT /users/{user}` — update user
- All routes restricted to owner role via middleware.

### Policies & Authorization

- User policy restricts all CRUD actions to the owner role only.

### Tests

- Assert owner can access all user management pages.
- Assert non-owner roles receive 403 on all user management routes.
- Assert a user is created with the correct role.
- Assert the last owner account cannot have its role changed.
- Assert role changes are logged in the audit trail.

## Acceptance Criteria

### Client-side

#### Layout

- The user management page is accessible from the Settings section in the sidebar navigation

#### Behavior

- The user list displays all users with their name, email, role, active status, and last login date
- Roles are displayed as color-coded badges for quick identification
- The owner can create a new user by filling in name, email, password, and selecting a role
- The owner can change a user's role and the change takes effect immediately
- The owner can toggle a user's active status (inactive users cannot log in)

#### Validation

- An error is displayed if the owner attempts to remove the last owner account
- An error is displayed for duplicate email addresses or missing required fields

---

### Server-side

#### Side-effects

- Only users with the Owner role can access user management — all other roles are denied
- Creating a user persists a new account with the selected role
- Changing a user's role takes effect immediately for their next action
- Deactivating a user prevents them from logging in
- The system prevents removing the last Owner account to avoid lockout
- Role changes are recorded in the audit trail
