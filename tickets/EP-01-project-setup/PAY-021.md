# [GW] Payroll - Create Database Seeders and Salary Components

## Description

Create the salary components table and all database seeders required to bootstrap the application with essential system data. This includes the company record, roles and permissions, the PHP base currency, 11 system salary components (earnings and deductions used in the payroll computation pipeline), 4 built-in bonus types, and an admin user. The master seeder orchestrates all seeders in dependency order so that a fresh database produces a fully functional starting state.

## Technical Implementation

### What Migrations

- Create the salary components table with ULID primary key, company reference, name, code, type (earning/deduction/benefit), taxability flag, system flag, description, sort order, and active flag.
- Composite unique constraint on company and code to prevent duplicate component codes within the same company.

### What Models

- Create SalaryComponent model in the Compensation domain with ULID support, factory, and scopes for filtering by active, system, custom, earnings, deductions, benefits, and ordered.
- System components cannot be deleted — the model provides a helper to check deletability.
- Factory includes state methods for system, earning, deduction, benefit, and inactive.

### Tests

- Assert the seeder runs without errors on a fresh database.
- Assert exactly one company exists after seeding.
- Assert 4 roles exist: owner, payroll_officer, finance, employee.
- Assert the PHP base currency is seeded with the correct base flag and rate.
- Assert 11 system salary components are seeded with correct types and taxability.
- Assert 4 system bonus types are seeded.
- Assert the admin user exists with the owner role and can authenticate.
- Assert re-running the seeder does not create duplicate records.
- Assert system salary components cannot be marked as deletable.
- Assert the unique constraint prevents duplicate component codes within the same company.

### Edge Cases & Notes

- Seeders must be idempotent — re-running should not create duplicates.
- ULIDs are generated at runtime, not hardcoded.
- The admin password ("password") is for development only.
- System components use low sort orders (1-21) so they appear first; custom components should use higher numbers.

## Acceptance Criteria

### Server-side

#### Side-effects

- Running the database seeder on a fresh database populates all required system data without errors
- The system starts with one company, 4 user roles, 1 base currency (Philippine Peso), 11 salary components, 4 bonus types, and one admin user
- The admin user can log in with the seeded credentials
- Salary components are categorized as either earnings or deductions with appropriate taxability settings
- System-provided salary components cannot be deleted through the application
- Running the seeder a second time does not create duplicate records
