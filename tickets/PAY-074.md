# PAY-074: In-App Notification System

## Description

Implement an in-app notification system using Laravel's built-in notification framework with the database channel. This includes creating the notifications table, a base notification infrastructure, a NotificationController for listing and marking notifications as read, and a frontend notification bell icon in the application header with an unread count badge and dropdown panel. The unread count is delivered via Inertia shared props so it is available on every page load.

## Technical Implementation

### Migrations
- **`database/migrations/xxxx_create_notifications_table.php`** — Laravel's default notifications table:
  - `id` (uuid, primary key)
  - `type` (string — notification class name)
  - `notifiable_type`, `notifiable_id` (morphs — the user receiving the notification)
  - `data` (JSON — notification payload)
  - `read_at` (nullable timestamp)
  - `created_at`, `updated_at` (timestamps)
  - Index on `notifiable_type`, `notifiable_id`
  - Index on `read_at` (for unread count queries)

### Models
- **`app/User/Models/User.php`** — Ensure the `Notifiable` trait is present (Laravel default), add:
  - `unreadNotificationCount(): int` — helper method returning `$this->unreadNotifications()->count()`

### Routes & Controllers
- **`GET /notifications`** — Route name: `notifications.index`
- **`PATCH /notifications/{notification}/read`** — Route name: `notifications.mark-as-read`
- **`POST /notifications/mark-all-read`** — Route name: `notifications.mark-all-read`
- **`app/Http/Controllers/Web/NotificationController`**
  - `index()`: returns paginated notifications for the authenticated user, ordered by `created_at` descending
  - `markAsRead(DatabaseNotification $notification)`: marks a single notification as read, redirects back
  - `markAllAsRead()`: marks all unread notifications as read for the authenticated user, redirects back
  - Middleware: `auth` on all routes

### Validation & DTOs
- **`app/Http/Requests/Notification/MarkNotificationReadRequest.php`** — Validates notification belongs to the authenticated user

### API Resources
- **`app/Http/Resources/NotificationResource.php`** — Formats notification for frontend:
  - `id`, `type` (human-readable label derived from class name)
  - `data` (the notification payload: title, message, action_url)
  - `read_at` (null if unread)
  - `created_at` (formatted with relative time)

### Shared Props (Inertia Middleware)
- **`app/Http/Middleware/HandleInertiaRequests.php`** — Add to `share()`:
  - `notifications.unread_count`: integer count of unread notifications for the authenticated user
  - `notifications.recent`: array of the 5 most recent notifications (for the dropdown preview)

### Frontend
- **`resources/js/components/layout/notification-bell.tsx`** — Header notification bell:
  - Bell icon with unread count badge (red circle with number, hidden when 0)
  - Clicking opens a dropdown panel
  - Dropdown shows recent notifications (from shared props)
  - Each notification: icon, title, message preview, relative timestamp
  - Unread notifications have a visual indicator (dot or background highlight)
  - Click on notification: marks as read via PATCH request, navigates to action_url if present
  - "Mark all as read" link at the top of the dropdown
  - "View all notifications" link at the bottom navigating to `/notifications`

- **`resources/js/pages/notifications/index.tsx`** — Full notifications page:
  - Lists all notifications for the current user, paginated
  - Each notification: title, message, timestamp, read/unread status
  - Click to mark as read and navigate
  - "Mark all as read" button
  - Empty state: "No notifications"

- **`resources/js/types/notification.ts`** — TypeScript types:
  - `Notification` type with id, type, data (title, message, action_url), read_at, created_at
  - `NotificationPageProps` type

### Tests
- **`tests/Feature/Notification/NotificationSystemTest.php`**
  - Authenticated user can view their notifications
  - Authenticated user can mark a notification as read
  - Authenticated user can mark all notifications as read
  - User cannot mark another user's notification as read
  - Unauthenticated user is redirected to login
  - Unread count is included in Inertia shared props
  - Recent notifications are included in Inertia shared props
  - Notifications are sorted by newest first
  - Pagination works correctly on the full notifications page

### Edge Cases & Notes
- The unread count query runs on every Inertia request via shared props; ensure it is efficient (indexed `read_at` column, count query only)
- The recent notifications in shared props should be limited to 5 items to keep the payload small
- Notification data JSON must follow a consistent structure (`title`, `message`, `action_url`) across all notification types
- Consider debouncing the mark-as-read request if the user clicks multiple notifications rapidly
