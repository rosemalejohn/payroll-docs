# [GW] Payroll - Configure Roles and Permissions with Spatie

## Description

Install and configure the Spatie permissions package to implement the four-role authorization system: owner, payroll_officer, finance, and employee. This ticket creates the role definitions, seeds them, and provides middleware and helper utilities for role checking throughout the application.

## Technical Implementation

### What Migrations

- Publish and run Spatie permission migrations creating the roles, permissions, and pivot tables.

### What Models

- Add the roles trait to the User model for role assignment and checking capabilities.

### Routes & Controllers

- Register role and permission middleware aliases for use in route definitions.
- Routes can use role-based middleware (e.g., restricting to owner or payroll_officer).

### Tests

- Assert all 4 roles exist after seeding.
- Assert a user can be assigned and checked for a role.
- Assert role middleware blocks unauthorized access (e.g., employee role cannot access owner-only routes).
- Assert the admin seeder creates a user with the owner role.
- Assert authenticated page responses include the user's roles.

### Environment & Config

- Publish and configure the Spatie permission config with team support disabled and appropriate cache settings.
- Ensure the User model guard matches the permission config guard.

### Edge Cases & Notes

- Roles are authorization labels only; granular permissions can be added later.
- The employee role is the most restrictive: read-only access to own finalized payslips only.
- Frontend role checks are UX-only — all enforcement must happen server-side.

## Acceptance Criteria

### Server-side

#### Side-effects

- Four roles exist in the system: Owner, Payroll Officer, Finance, and Employee
- Users can be assigned a role and the system correctly identifies their role
- Routes protected by role restrictions deny access to unauthorized users with a 403 response
- The admin user is created with the Owner role during initial setup
- Role information is shared with the frontend for display purposes, but all access enforcement happens on the server
