# PAY-050: RecomputePayrollRunAction - Batch Recompute All Payslips

## Description

Implement a batch recomputation action that recalculates all payslips in a payroll run from scratch. This action deletes all existing line items, re-runs `ComputePayslipAction` for every employee in the run, and recalculates the run-level totals. It is only allowed on draft runs and must be idempotent (same inputs always produce the same outputs). This action is triggered whenever data changes that affect payslip computation.

## Technical Implementation

### Action

**`app/Payroll/Actions/RecomputePayrollRunAction.php`**

```php
class RecomputePayrollRunAction
{
    public function handle(PayrollRun $payrollRun): void
    {
        // 1. Guard: only draft runs
        if (!$payrollRun->state->equals(DraftState::class)) {
            throw new InvalidRunStateException(
                "Cannot recompute a payroll run in '{$payrollRun->state}' state. Only draft runs can be recomputed."
            );
        }

        // 2. Delete all existing line items for the run
        PayslipLineItem::whereIn(
            'payslip_id',
            $payrollRun->payslips()->pluck('id')
        )->delete();

        // 3. Recompute each payslip
        $computeAction = app(ComputePayslipAction::class);

        $payrollRun->payslips->each(function (Payslip $payslip) use ($computeAction) {
            // Reset computed fields before recomputation
            $payslip->update([
                'gross_pay'              => 0,
                'total_bonuses'          => 0,
                'total_deductions'       => 0,
                'net_pay'                => 0,
                'net_pay_floored'        => false,
                'absent_deduction'       => 0,
                'sss_contribution'       => 0,
                'philhealth_contribution' => 0,
                'pagibig_contribution'   => 0,
                'hmo_deduction'          => 0,
                'mp2_deduction'          => 0,
                'sss_wisp_deduction'     => 0,
                'cash_advance_deduction' => 0,
                'bir_tax'                => 0,
                'computed_at'            => null,
            ]);

            $computeAction->handle($payslip);
        });

        // 4. Recalculate run-level totals
        $payrollRun->update([
            'total_gross'      => $payrollRun->payslips()->sum('gross_pay'),
            'total_deductions' => $payrollRun->payslips()->sum('total_deductions'),
            'total_net'        => $payrollRun->payslips()->sum('net_pay'),
            'total_bonuses'    => $payrollRun->payslips()->sum('total_bonuses'),
            'employee_count'   => $payrollRun->payslips()->count(),
        ]);
    }
}
```

### Triggers

This action is called by other parts of the system when data affecting computation changes:

| Trigger | Source |
|---|---|
| Absent days updated | `UpdatePayslipAbsentDaysAction` (PAY-043) |
| Bonus added/removed/modified on the run | Bonus management actions (PAY-045) |
| Employee added to the run | `AddEmployeeToRunAction` (PAY-042) |
| Employee removed from the run | `RemoveEmployeeFromRunAction` (PAY-042) |
| Employee compensation updated | Compensation sync or manual update |
| Payroll run submitted | `SubmitPayrollRunAction` (PAY-041) -- final recompute before state change |

For single-employee changes (e.g., absent days), the trigger may optionally call `ComputePayslipAction` for just that employee and then recalculate run totals, rather than recomputing all payslips. The full batch recompute is used when changes affect multiple employees or when a clean slate is needed.

### Database Transaction

The entire operation runs inside a database transaction to ensure atomicity:

```php
DB::transaction(function () use ($payrollRun) {
    // delete line items, recompute payslips, update totals
});
```

If any single payslip computation fails, the entire operation rolls back.

### Performance Considerations

- For large runs (500+ employees), consider chunking the payslip iteration: `$payrollRun->payslips()->chunk(100, fn ($chunk) => ...)`.
- Line item deletion uses a bulk `whereIn` query rather than iterating per payslip.
- Run-level totals use aggregate SQL queries (`SUM`, `COUNT`) rather than PHP-side summation.

### Edge Cases

- Run with zero employees: action completes successfully, all totals set to 0.
- Single payslip computation failure: entire transaction rolls back, error is thrown.
- Concurrent recomputation: database transaction provides isolation; second call waits or fails with lock timeout.

## Acceptance Criteria

- [ ] All payslips in the run are recomputed from scratch (line items deleted and regenerated).
- [ ] Run-level totals (`total_gross`, `total_deductions`, `total_net`, `total_bonuses`, `employee_count`) are updated after recomputation.
- [ ] Action only works on draft runs; non-draft runs throw `InvalidRunStateException`.
- [ ] Line items are fully regenerated (old items deleted, new items created).
- [ ] Action is idempotent: running it twice with the same input data produces identical results.
- [ ] The operation is wrapped in a database transaction for atomicity.
- [ ] A failure in any single payslip computation rolls back the entire operation.
- [ ] Run totals use aggregate SQL queries for accuracy.
- [ ] Action is triggered by absent days changes, bonus changes, employee add/remove, and compensation updates.
- [ ] Zero-employee runs complete without error, setting all totals to 0.
