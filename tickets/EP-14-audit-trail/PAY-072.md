# [GW] Payroll - Audit Trail Page with Filters

## Description

Build a dedicated audit trail page accessible only to the `owner` role. This page displays all audit log entries from the `audits` table, showing timestamp, acting user, model type, action (created/updated), and old/new values in a readable format. The page includes filters for payroll run, date range, user, and model type to help owners quickly locate specific changes. The audit trail is strictly read-only — audits cannot be edited or deleted through the UI.

## Technical Implementation

### Actions
- **`app/Audit/Actions/ListAuditsAction.php`** — Builds a paginated, filtered query on the `audits` table:
  - Accepts filters: `model_type`, `user_id`, `date_from`, `date_to`, `payroll_run_id`
  - Eager loads `user` relationship (actor)
  - Orders by `created_at` descending (newest first)
  - Returns paginated results (25 per page)

### Queries
- **`app/Audit/Queries/FilterAuditsQuery.php`** — Encapsulates complex filtering logic:
  - Filter by `auditable_type` (maps friendly names like "PayrollRun" to fully qualified class names)
  - Filter by `user_id` for actor filtering
  - Filter by `created_at` date range
  - Filter by `payroll_run_id` — joins through auditable when `auditable_type` is PayrollRun, or filters payslip audits by their `payroll_run_id`

### Routes & Controllers
- **`GET /audit-trail`** — Route name: `audit-trail.index`
- **`app/Http/Controllers/Web/AuditController@index`**
  - Middleware: `auth`, `role:owner`
  - Authorizes via `AuditPolicy@viewAny`
  - Passes paginated audits, filter options (users list, model types, payroll runs), and active filters as Inertia props
  - Returns `Inertia::render('audit-trail/index')`

### Validation & DTOs
- **`app/Http/Requests/Audit/IndexAuditRequest.php`** — Validates filter parameters:
  - `model_type`: nullable, string, in allowed model types
  - `user_id`: nullable, string, exists in users table
  - `date_from`: nullable, date
  - `date_to`: nullable, date, after_or_equal:date_from
  - `payroll_run_id`: nullable, string, exists in payroll_runs table

### API Resources
- **`app/Http/Resources/AuditResource.php`** — Formats audit entries for the frontend:
  - `id`, `event`, `auditable_type` (human-readable), `auditable_id`
  - `old_values`, `new_values` (formatted key-value pairs)
  - `user` (actor name and role)
  - `created_at` (formatted timestamp)
  - `url`, `ip_address`

### Policies & Authorization
- **`app/Audit/Policies/AuditPolicy.php`**
  - `viewAny()`: only `owner` role returns true
  - No `create`, `update`, `delete` methods — audits are immutable

### Frontend
- **`resources/js/pages/audit-trail/index.tsx`** — Audit trail page:
  - Filter bar: model type dropdown, user dropdown, date range pickers, payroll run selector
  - Data table with columns: Timestamp, User, Action, Model, Changes (old → new)
  - Changes column renders a diff-style display showing field name, old value (red), new value (green)
  - Pagination controls
  - No edit or delete actions anywhere on the page
  - Empty state when no audits match filters

- **`resources/js/components/audit/audit-changes-cell.tsx`** — Renders old/new value pairs in a compact diff format
- **`resources/js/types/audit.ts`** — TypeScript types for Audit, AuditFilters

### Tests
- **`tests/Feature/Audit/AuditTrailPageTest.php`**
  - Owner can access the audit trail page
  - Non-owner roles (payroll_officer, finance, employee) receive 403
  - Unauthenticated user is redirected to login
  - Filter by model type returns only matching audits
  - Filter by user returns only audits by that actor
  - Filter by date range returns audits within range
  - Filter by payroll run returns related audits
  - Results are paginated at 25 per page
  - Results are sorted by newest first
  - Audit entries display correct old/new values

### Edge Cases & Notes
- Model type filter must map user-friendly names (e.g., "Payroll Run") to fully qualified class names (e.g., `App\Payroll\Models\PayrollRun`)
- Old/new values are stored as JSON; handle gracefully if values are null or malformed
- Large audit tables may need database indexing optimization; ensure composite index on `(auditable_type, created_at)` for common queries
- The page must remain performant even with thousands of audit records — rely on server-side pagination, never load all records

## Acceptance Criteria

- [ ] Audit trail page is accessible at `/audit-trail` for owner role only
- [ ] Non-owner roles receive a 403 Forbidden response
- [ ] Audit entries display timestamp, actor (user name), model type, action, and old/new values
- [ ] Filter by model type works correctly (PayrollRun, Payslip, EmployeeCompensation, Currency, Employee)
- [ ] Filter by date range returns audits within the specified period
- [ ] Filter by user returns audits performed by the selected actor
- [ ] Filter by payroll run returns all audits related to that run
- [ ] Results are sorted by timestamp, newest first
- [ ] Results are paginated (25 per page)
- [ ] Audit entries are read-only — no edit or delete actions available
- [ ] State transitions and value changes are displayed in a readable diff format
- [ ] Page performs well with large datasets via server-side pagination
