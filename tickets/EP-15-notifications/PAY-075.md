# [GW] Payroll - Payroll Workflow Notifications

## Description

Implement notification classes for each step in the payroll workflow. When a payroll run transitions between states, the appropriate users are notified via the in-app notification system (PAY-074). Finance users are notified when a run is submitted for review, owners when a run is approved, payroll officers when a run is finalized or returned, and employees when their payslips are available.

## Technical Implementation

### What Actions

- Send Payroll Run Submitted Notification action: sends notification to all finance users when a run transitions from draft to submitted. Includes run name, period, and link to the run.
- Send Payroll Run Approved Notification action: sends notification to all owner users when finance approves a run. Includes run name, period, approver name, and link to the run.
- Send Payroll Run Finalized Notification action: sends notification to the payroll officer who originally submitted the run when the owner finalizes it. Includes run name, period, finalizer name, and link to the run.
- Send Payslip Available Notification action: sends notification to each employee with a payslip in the finalized run. Includes period and link to their payslip.
- Send Payroll Run Returned Notification action: sends notification to appropriate users when a run is returned to a previous state. Recipients depend on the previous state — finance users if returned from under_review, or the submitting payroll officer if returned from submitted. Includes run name, period, new state, actor name, and reason.

### Events & Listeners

- PayrollRunSubmitted event → listener queries all finance users and sends submitted notification.
- PayrollRunApproved event → listener queries all owner users and sends approved notification.
- PayrollRunFinalized event → listener sends finalized notification to the submitting payroll officer, and dispatches a queued job for bulk payslip notifications to employees.
- PayrollRunReturned event → listener determines recipients based on previous state and sends returned notification.

### Jobs

- Send Bulk Payslip Notifications job: queued job that iterates through all payslips in a finalized run and sends payslip available notification to each employee. Dispatched to the notifications queue.

### Tests

- Assert submitting a run sends notification to all finance users.
- Assert approving a run sends notification to all owner users.
- Assert finalizing a run sends notification to the submitting payroll officer.
- Assert finalizing a run sends payslip notifications to all employees in the run.
- Assert returning a run to draft sends notification to appropriate users based on previous state.
- Assert notification data includes run name, period, and action URL.
- Assert notifications appear in the recipient's notification bell dropdown.
- Assert users who are not recipients do not receive notifications.
- Assert payslip notifications are dispatched to the queue (not synchronous).

### Edge Cases & Notes

- For large payroll runs (hundreds of employees), payslip notifications must be queued to avoid timeouts during finalization.
- If a user has multiple roles (e.g., owner + finance), they should not receive duplicate notifications for the same event.
- The returned notification must handle multiple possible previous states and route to the correct recipients.
- If an employee does not have a user account (no login access), their payslip notification is skipped silently.
- Notification content must include enough context (run name, period) so the recipient understands without clicking through.

## Acceptance Criteria

### Client-side

#### Behavior

- Finance users see a notification in their bell icon when a payroll run is submitted for review
- Owner users see a notification when a payroll run is approved and ready for finalization
- The payroll officer who submitted a run sees a notification when it is finalized
- Employees see a "Payslip Available" notification when their payslip is finalized
- Users see a notification when a run they are involved with is returned to a previous state
- Each notification includes the run name and period dates for context
- Clicking a notification navigates to the relevant payroll run or payslip page

---

### Server-side

#### Side-effects

- Notifications are sent automatically when payroll runs transition between states
- Only the appropriate role recipients receive each notification type
- Users with multiple roles do not receive duplicate notifications for the same event
- Payslip notifications for employees are dispatched to a background queue for performance
- Employees without user accounts are skipped without errors
- Notification data includes the run name, period, actor name, and a link to the relevant page
