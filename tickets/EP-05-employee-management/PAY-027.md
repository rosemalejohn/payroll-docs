# [GW] Payroll - Employee Create/Edit Page

## Description

Build the employee create and edit pages with full server-side validation, action classes for persistence, and Inertia React page components using the reusable form components. Successful mutations redirect to the employee list with a flash message.

## Technical Implementation

### What Actions

- Create Employee action: creates a new employee record with a generated ULID, sets the company from the authenticated user, generates a local HRIS ID for manually-created employees, and initializes default payroll flags.
- Update Employee action: updates allowed fields on the employee record. Does not overwrite payroll-specific flags (managed separately in PAY-029).

### Validation & DTOs

- Store Employee request: validates employee number (required, unique), first name (required), last name (required), middle name (optional), email (optional), department (optional), position (optional), employment type (required, must be regular/contractual/probationary), employment status (required), date hired (optional, cannot be in the future), and pay basis (required, must be monthly/daily).
- Update Employee request: same rules, but employee number uniqueness ignores the current employee.

### Routes & Controllers

- `GET /employees/create` — create form
- `POST /employees` — store new employee
- `GET /employees/{employee}/edit` — edit form with pre-filled data
- `PUT /employees/{employee}` — update employee
- All routes restricted to Owner and Payroll Officer.

### Policies & Authorization

- Route middleware and form request authorization restrict access to Owner and Payroll Officer.

### Tests

- Assert Owner and Payroll Officer can access create and edit pages.
- Assert Employee and Finance roles receive 403.
- Assert successful creation with valid data redirects to employee list with success message.
- Assert validation fails for missing required fields, duplicate employee number, and future hire date.
- Assert edit page pre-fills existing employee data.
- Assert successful update redirects with success message.
- Assert employee number uniqueness on edit ignores the current employee.

### Edge Cases & Notes

- The create page must not show payroll-specific flags — those are managed in PAY-029.
- HRIS employee ID is auto-generated for manually created employees and should not be editable.
- Middle name is optional and handled gracefully as null.

## Acceptance Criteria

### Client-side

#### Layout

- The create page is accessible via the "Add Employee" button on the employee list
- The edit page is accessible from the employee list actions column

#### Behavior

- The create form has sections for Personal Information (employee number, name, email) and Employment Details (department, position, type, status, hire date, pay basis)
- The edit form is pre-filled with the existing employee's data
- After successfully saving, the user is redirected to the employee list with a success message
- The Cancel button returns to the employee list without saving
- The Submit button shows a loading state and is disabled while processing

#### Validation

- Missing required fields (employee number, first name, last name) display inline error messages
- Duplicate employee numbers display a validation error
- Future hire dates display a validation error
- Invalid employment type or pay basis values display a validation error

---

### Server-side

#### Side-effects

- Creating an employee persists a new record with the selected fields and default payroll settings
- Updating an employee modifies only the allowed fields without affecting payroll-specific flags
- Only Owner and Payroll Officer roles can create or edit employees
