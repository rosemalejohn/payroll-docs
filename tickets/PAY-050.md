# [GW] Payroll - RecomputePayrollRunAction - Batch Recompute All Payslips

## Description

To ensure both data integrity and application performance, the payroll system uses two recomputation strategies: a **full recomputation** for complete refreshes and a **granular recomputation** for targeted, efficient updates. This ticket defines the actions for both methods.

1.  **Full Recomputation (`RecomputePayrollRunAction`):** A brute-force action that recalculates every payslip in a run. It is safe but can be slow on large runs. It is reserved for critical checkpoints like pre-submission.
2.  **Granular Recomputation (`RecomputeAffectedPayslipsAction`):** A performant action that recomputes only a specific subset of payslips affected by a change. This is the default method for most edits in a draft run.

## Technical Implementation

### 1. Full Recomputation Strategy (The "Safety Net")

#### Action: `app/Payroll/Actions/RecomputePayrollRunAction.php`

This action performs a full, clean-slate recomputation of every payslip in a run.

```php
class RecomputePayrollRunAction
{
    public function handle(PayrollRun $payrollRun): void
    {
        // 1. Guard: only draft runs are allowed
        if (!$payrollRun->state->equals(DraftState::class)) {
            throw new InvalidRunStateException("Only draft runs can be recomputed.");
        }

        DB::transaction(function () use ($payrollRun) {
            // 2. Delete all existing line items for the entire run
            PayslipLineItem::whereIn('payslip_id', $payrollRun->payslips()->pluck('id'))->delete();

            // 3. Recompute each payslip from scratch
            $computeAction = app(ComputePayslipAction::class);
            $payrollRun->payslips->each(fn (Payslip $p) => $computeAction->handle($p));

            // 4. Recalculate all run-level totals from the database
            $payrollRun->update([
                'total_gross'      => $payrollRun->payslips()->sum('gross_pay'),
                'total_deductions' => $payrollRun->payslips()->sum('total_deductions'),
                'total_net'        => $payrollRun->payslips()->sum('net_pay'),
            ]);
        });
    }
}
```

---

### 2. Granular Recomputation Strategy (The "Fast Path")

This performant strategy targets only specific payslips and updates run totals efficiently using adjustment values.

#### Action: `app/Payroll/Actions/RecomputeAffectedPayslipsAction.php`

-   **Accepts:** `PayrollRun $payrollRun`, `Collection|array $payslipIds`.
-   **Logic:** Iterates only over the specified payslip IDs. For each, it calculates the difference (adjustment) between the old and new payslip totals and dispatches `UpdatePayrollRunTotalsAction` with the adjustment values.

#### Action: `app/Payroll/Actions/UpdatePayrollRunTotalsAction.php`

-   **Accepts:** `PayrollRun $payrollRun`, `array $adjustments`.
-   **Logic:** Atomically updates the `payroll_runs` totals using `DB::table()->increment/decrement()` with the provided adjustment values. This avoids slow, full-table `SUM()` queries.
-   **Example adjustment:** `['total_gross' => 150.75, 'total_net' => -50.25]`

### Triggers and Usage

-   **Use Full Recomputation (`RecomputePayrollRunAction`):**
    -   Primarily used just before a payroll run is submitted for review (`SubmitPayrollRunAction`). This acts as a final consistency check and ensures all calculations are perfectly in sync.

-   **Use Granular Recomputation (`RecomputeAffectedPayslipsAction`):**
    -   **Absent days update:** When one employee's absent days change, recompute only their payslip.
    -   **Bonus changes:** When a bonus is changed, recompute only the payslips of employees affected by that bonus.
    -   **Currency rate update:** When a currency rate changes, recompute only the payslips of employees paid in that currency.
    -   **Compensation update:** When one employee's salary is updated, recompute only their payslip.

### Performance Considerations

-   For large runs (500+ employees), granular recomputation is significantly faster as it avoids iterating through and re-calculating hundreds of unaffected payslips.
-   The `UpdatePayrollRunTotalsAction` is critical for performance, as it prevents full table scans on the `payslips` table for every minor adjustment.

## Acceptance Criteria

-   [ ] `RecomputePayrollRunAction` correctly recomputes all payslips in a run and updates totals.
-   [ ] `RecomputeAffectedPayslipsAction` correctly recomputes only the specified subset of payslips.
-   [ ] `UpdatePayrollRunTotalsAction` correctly updates run totals using adjustment values.
-   [ ] Updating a single employee's data (e.g., absent days) triggers the granular action, not the full one.
-   [ ] Submitting a payroll run for review triggers the full recomputation action to ensure consistency.
-   [ ] Both actions are blocked for payroll runs that are not in the `draft` state.
-   [ ] The system's performance is improved when making isolated changes to a large payroll run.
