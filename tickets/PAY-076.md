# [GW] Payroll - Duplicate Payroll Run Prevention

## Description

Prevent the creation of duplicate payroll runs for the same pay period. When a payroll officer or owner attempts to create a new run, the system must validate that no existing run for the same period exists in `draft`, `submitted`, `under_review`, or `finalized` state. If a duplicate is found, the system returns a clear validation error naming the existing run and its current status. Cancelled runs do not block the creation of new runs. Additionally, the payroll run creation form displays recent runs for the selected period to proactively help users avoid duplicates.

## Technical Implementation

### Actions
- **`app/Payroll/Actions/CreatePayrollRunAction.php`** â€” Add duplicate detection before creating a run:
  - Query `payroll_runs` table for matching `period_start` and `period_end` where `status` is NOT `cancelled`
  - If a match is found: throw a `ValidationException` with message: `"A payroll run for this period already exists: '{run_name}' (Status: {status}). Please use the existing run or cancel it before creating a new one."`
  - If no match: proceed with creation

### Queries
- **`app/Payroll/Queries/FindOverlappingPayrollRunsQuery.php`** â€” Detects overlapping runs:
  - Accepts `period_start` and `period_end` dates
  - Returns payroll runs where the period overlaps (not just exact match) and status is not `cancelled`
  - Overlap logic: `existing.period_start <= new.period_end AND existing.period_end >= new.period_start`
  - Returns collection with `id`, `name`, `period_start`, `period_end`, `status`

### Validation & DTOs
- **`app/Http/Requests/Payroll/StorePayrollRunRequest.php`** â€” Add custom validation rule:
  - `period_start`: required, date
  - `period_end`: required, date, after_or_equal:period_start
  - Custom rule: `new NoOverlappingPayrollRunRule($this->period_start, $this->period_end)` â€” uses `FindOverlappingPayrollRunsQuery`

### Rules
- **`app/Payroll/Rules/NoOverlappingPayrollRunRule.php`** â€” Custom validation rule:
  - Implements `Illuminate\Contracts\Validation\ValidationRule`
  - Uses `FindOverlappingPayrollRunsQuery` to check for overlaps
  - Failure message includes the existing run name and status

### Routes & Controllers
- **`GET /payroll-runs/create`** â€” Existing route, modified to include recent runs data
- **`app/Http/Controllers/Web/PayrollRunController@create`** â€” Add to Inertia props:
  - `recent_runs`: list of payroll runs from the last 3 months with `id`, `name`, `period_start`, `period_end`, `status`

### API Resources
- **`app/Http/Resources/RecentPayrollRunResource.php`** â€” Slim resource for the creation form:
  - `id`, `name`, `period_start`, `period_end`, `status`, `created_at`

### Frontend
- **`resources/js/pages/payroll-runs/create.tsx`** â€” Enhance the creation form:
  - After user selects period dates, display a "Recent Runs" info panel below the date fields
  - Panel shows any existing runs that overlap with the selected period
  - If overlapping run found: show warning with run name, status, and link to the existing run
  - If no overlap: show a green checkmark "No existing runs for this period"
  - Client-side check is UX-only; server validation is the source of truth
  - Validation error from server displayed inline with the period fields

- **`resources/js/components/payroll/recent-runs-panel.tsx`** â€” Component showing recent/overlapping runs:
  - Accepts `recent_runs` and `selected_period` props
  - Filters and highlights overlapping runs
  - Each run entry: name, period, status badge, link to view

### Tests
- **`tests/Feature/Payroll/DuplicatePayrollRunPreventionTest.php`**
  - Creating a run with an overlapping period returns 422 with descriptive error when a draft run exists
  - Creating a run with an overlapping period returns 422 when a submitted run exists
  - Creating a run with an overlapping period returns 422 when an under_review run exists
  - Creating a run with an overlapping period returns 422 when a finalized run exists
  - Creating a run for a period with only a cancelled run succeeds
  - Creating a run for a completely different period succeeds
  - Validation error message includes the existing run name and status
  - Partial period overlap is detected (new run partially overlaps existing run)
  - Recent runs are returned in the create form Inertia props

### Edge Cases & Notes
- Period overlap detection must handle partial overlaps, not just exact date matches â€” a run for March 1-15 should block a run for March 10-20
- Cancelled runs must be explicitly excluded from the overlap check
- Race condition: two users creating runs for the same period simultaneously â€” consider adding a database-level unique constraint or advisory lock on (period_start, period_end, non-cancelled status)
- The recent runs panel on the creation form should update dynamically as the user changes the period dates (client-side filtering of `recent_runs` prop)

## Acceptance Criteria

- [ ] Creating a payroll run for a period that overlaps an existing finalized run returns a validation error

- [ ] Validation error message includes the existing run's name and current status

- [ ] Payroll run creation form shows recent runs for the selected period
