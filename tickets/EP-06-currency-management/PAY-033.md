# [GW] Payroll - Currency Management Settings

## Description

Build the currency management settings page where users can view, add, update, and deactivate currencies. The currencies table stores exchange rates as "1 [FOREIGN] = X PHP" with high decimal precision. PHP (Philippine Peso) is the immutable base currency. The currency snapshot table captures all active currency rates at the time a payroll run is created, ensuring historical integrity. Non-owner roles can view currencies in read-only mode. The Owner can add new currencies, update exchange rates (with warnings for open payroll runs), and deactivate currencies with safety checks preventing deactivation when employees still use them.

## Technical Implementation

### What Migrations

- Create the currencies table with ULID primary key, company reference, code (ISO 4217, unique per company), name, symbol, rate (high-precision decimal for exchange rates), base currency flag, active flag, rate update timestamp, and standard timestamps.
- Create the payroll run currency snapshots table with ULID primary key, payroll run reference, currency reference, denormalized currency code, snapshot rate, and snapshot timestamp.

### What Models

- Create Currency model in the Currency domain with ULID support, factory, conversion helpers (to PHP and from PHP), and audit trail integration. Factory includes states for base currency and inactive.
- Create PayrollRunCurrencySnapshot model in the Payroll domain with ULID support and factory.

### What Actions

- Create Currency action: validates the code is unique per company, rate is positive, sets the currency as non-base and active, records the rate update timestamp.
- Update Currency Rate action: validates the new rate is positive, prevents updating the base currency (PHP), updates the rate and rate update timestamp, and checks for open payroll runs with employees in this currency to return a warning if affected runs exist.
- Deactivate Currency action: prevents deactivating the base currency, checks for active employees with compensation in this currency, returns an error with count if employees are found, and sets the currency to inactive if safe.

### Validation & DTOs

- Store Currency request: validates code (required, uppercase, unique per company), name (required), symbol (required), and rate (required, must be greater than zero).
- Update Currency Rate request: validates rate (required, must be greater than zero).

### Routes & Controllers

- `GET /settings/currencies` — paginated currency list with rate, last updated, and active status. Owner sees edit and toggle buttons; non-owner sees read-only table.
- `POST /settings/currencies` — create new currency (Owner only).
- `PUT /settings/currencies/{currency}` — update currency rate (Owner only).
- `PATCH /settings/currencies/{currency}/toggle-active` — toggle active status (Owner only).

### Policies & Authorization

- All authenticated users can view currencies.
- Only the Owner role can create currencies, update rates, or toggle active status.

### Tests

- Assert all authenticated users can view the currency list.
- Assert only Owner sees action buttons.
- Assert PHP base currency has no edit/toggle controls.
- Assert currency model uses ULID primary keys.
- Assert unique constraint on company and code prevents duplicates.
- Assert conversion helpers calculate correctly.
- Assert base currency rate is always 1.0 and immutable.
- Assert currency snapshots can be created and linked to payroll runs.
- Assert currency is created with correct data.
- Assert duplicate code per company is rejected.
- Assert rate of zero or negative is rejected.
- Assert non-owner roles receive 403 for create, update, and toggle.
- Assert new currency is immediately available in employee compensation currency selector.
- Assert rate updates correctly.
- Assert base currency (PHP) update is blocked.
- Assert audit log captures old and new rate values.
- Assert warning is returned when open payroll runs have employees in this currency.
- Assert deactivation is blocked when active employees use the currency (with employee count in error message).
- Assert PHP base currency cannot be deactivated.
- Assert deactivated currency is hidden from compensation currency selector.
- Assert existing payslips are unaffected by deactivation (they use snapshot data).
- Assert currency can be reactivated (toggled back to active).

### Edge Cases & Notes

- Rates are stored with 6 decimal places for precision.
- PHP base currency cannot be edited, deactivated, or deleted.
- The rate format convention is "1 [FOREIGN] = X PHP".

## Acceptance Criteria

### Client-side

#### Layout

- Accessible from Settings sidebar section

#### Behavior

- List shows code, name, symbol, rate, updated, status
- Owner sees Edit Rate and Toggle buttons per row
- Non-owner roles see read-only table only
- PHP row always present, no edit/toggle controls
- Rates displayed as "1 [CODE] = ₱[rate]"
- "Add Currency" button visible to Owner only
- Add modal has Code, Name, Symbol, Exchange Rate
- Live preview shows rate formula while typing
- Edit modal shows name/code read-only, editable rate
- "Last updated" timestamp shown for reference
- Warning lists affected open runs before saving
- Toggle button activates/deactivates currency per row
- Blocked deactivation shows affected employee count
- Safe deactivation shows "Deactivate [CODE]?" confirmation
- PHP row has no toggle button

#### Validation

- Error on duplicate currency code
- Error on zero or negative rate
- PHP rate cannot be edited

---

### Server-side

#### Side-effects

- PHP is base currency, rate fixed at 1.0
- Currency codes unique per company
- Audit trail tracks rate changes with old/new values
- Run creation snapshots all active currency rates
- Snapshot rates are immutable after creation
- New currencies created as active, non-base
- New currency available in compensation currency selector
- Only Owner can add currencies
- Exchange rate updates immediately on save
- Rate update timestamp recorded
- Warning returned for affected open payroll runs
- Deactivation blocked if employees use the currency
- PHP can never be deactivated
- Deactivated currencies hidden from currency selector
- Deactivated currencies excluded from future snapshots
- Existing payslips unaffected by deactivation
- Deactivated currencies can be reactivated anytime
