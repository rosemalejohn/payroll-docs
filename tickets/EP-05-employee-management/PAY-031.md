# [GW] Payroll - HRIS Employee Sync Job

## Description

Implement a daily background job that syncs employee data from the external HRIS application via API. The sync creates new employees and updates existing ones using the HRIS employee ID as the matching key, while preserving payroll-specific local fields that must never be overwritten by the sync.

## Technical Implementation

### What Actions

- Sync Employee From HRIS action: handles individual employee upsert logic — creates or updates the employee record using the HRIS employee ID as the matching key.

### Jobs

- HRIS Sync job: queued on Redis, makes an HTTP request to the HRIS API endpoint, iterates through the response, and processes each employee through the sync action.
- Scheduled to run daily at 02:00.
- Retry up to 3 times with exponential backoff on API connection failure.

### Commands

- Register the sync job in the scheduler to run daily at 02:00.

### Events & Listeners

- Dispatch a sync completed event with summary counts (created, updated, failed).

### Tests

- Assert new HRIS employees are created locally with correct data.
- Assert existing employees have synced fields updated without overwriting local payroll fields.
- Assert payroll-specific fields (pay basis, external salary, MP2, SSS WISP, HMO, government IDs) are preserved on sync.
- Assert the last synced timestamp is updated on each synced employee.
- Assert API failures are logged and the job retries.
- Assert individual employee parse errors are logged and skipped without halting the entire sync.

### Environment & Config

- Add HRIS API URL and token to the environment configuration.

### Edge Cases & Notes

- Synced fields (overwritten): employee number, first name, last name, middle name, email, department, position, employment type, employment status, date hired.
- Local-only fields (NEVER overwritten): pay basis, external salary flag, MP2 flag, SSS WISP flag, HMO amount, SSS number, PhilHealth number, Pag-IBIG number, BIR TIN.
- Individual employee errors should not halt the entire sync — log and continue.

## Acceptance Criteria

### Server-side

#### Side-effects

- The sync job runs daily at 02:00 and retrieves employee data from the HRIS system
- New employees found in the HRIS system are created locally with default payroll settings
- Existing employees have their HRIS-managed fields updated (name, department, position, status, etc.)
- Payroll-specific fields that are managed locally are NEVER overwritten by the sync (external salary, MP2, SSS WISP, HMO amount, government ID numbers, pay basis)
- Each synced employee's "last synced" timestamp is updated
- If the HRIS API is unavailable, the job retries up to 3 times before failing
- If an individual employee record fails to sync, the error is logged and the remaining employees continue to sync
- A summary of the sync results (created, updated, failed counts) is logged upon completion
