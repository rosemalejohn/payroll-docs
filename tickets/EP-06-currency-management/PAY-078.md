# [GW] Payroll - Currency Rate Update Mid-Run Warning

## Description

When the Owner updates a currency exchange rate, the system detects if any open payroll runs contain employees paid in that currency. If affected runs exist, the Owner is presented with a warning and given two options: (1) apply the new rate to current runs by re-snapshotting and recomputing affected payslips, or (2) keep the existing snapshot rate for current runs and apply the new rate only to future runs. All decisions are logged in the audit trail.

## Technical Implementation

### What Actions

- Enhanced Update Currency Rate action: before updating, checks for open runs (draft/submitted/under review) with employees in this currency. Returns affected runs list for the warning dialog if found.
- Apply Rate to Open Run action: updates the currency snapshot record for the run, recomputes all affected payslips, and adds a note on the run.
- Recompute Affected Payslips action: finds payslips in the run using the updated currency and re-runs the computation with the new snapshot rate.

### What Migrations

- Add a currency rate notes field (nullable text) to the payroll runs table for tracking rate change history on runs.

### Queries

- Find Affected Payroll Runs query: identifies open payroll runs containing employees with compensation in the given currency.

### Validation & DTOs

- DTO for the rate update decision: currency ID, new rate, selected run IDs to apply to, and decision type (apply to current or future only).
- Request validation: rate must be positive, decision required when affected runs exist, run IDs validated when applying to current runs.

### Routes & Controllers

- `PUT /currencies/{currency}/rate` — initial rate update check.
- `POST /currencies/{currency}/rate/apply-to-runs` — apply rate to selected open runs.
- Both routes restricted to Owner role.

### Tests

- Assert updating rate with no open runs proceeds without warning.
- Assert updating rate with affected runs returns warning with run list.
- Assert "apply to current" re-snapshots and recomputes affected payslips.
- Assert "future only" does not change existing snapshots.
- Assert recomputed payslips have correct values based on new rate.
- Assert banner appears on run detail page after rate is applied.
- Assert audit trail captures old rate, new rate, affected runs, and user decision.
- Assert recomputation is wrapped in a transaction for atomicity.

### Edge Cases & Notes

- If a run is under review and the rate is applied, finance should be alerted that figures have changed.
- The snapshot must be updated before recomputing payslips — ordering matters.
- If recomputation fails, the entire operation should roll back.
- PHP (base currency) rate changes should be blocked.

## Acceptance Criteria

### Client-side

#### Behavior

- Warning dialog lists affected runs with employee counts
- "Apply to Current Runs" recomputes affected payslips
- "Future Runs Only" preserves existing snapshots
- Info banner on run detail after rate applied

---

### Server-side

#### Side-effects

- "Apply" updates snapshot and recomputes affected payslips
- "Future Only" preserves existing snapshot rates
- Audit trail captures rate change decisions and values
- Recomputation is atomic — failure rolls back everything
- Only Owner can update currency rates
- PHP base currency rate cannot change
