# [GW] Payroll - Submit Payroll Run for Review

## Description

Implement the action and UI for submitting a payroll run from draft to submitted state. Before submission, the system performs pre-submit validation (run has employees, all payslips computed, no unresolved flags) and triggers a full recomputation. A confirmation dialog presents a summary (employee count, total gross, total net) before the user confirms. After submission, finance role users receive a notification.

## Technical Implementation

### What Actions

- Submit Payroll Run action: validates the run is in draft state, validates the run has at least one payslip, validates all payslips are computed (no payslip with zero gross and net unless the employee has external salary), validates no unresolved flags exist on any payslip, triggers full recomputation before the state transition, transitions state from draft to submitted, sets submitted by and submitted at fields, and dispatches a notification to all finance role users.

### Validation & DTOs

- Submit Payroll Run request: validates authorization (user is Owner or Payroll Officer) and that the run is in draft state.

### Routes & Controllers

- `POST /payroll-runs/{payrollRun}/submit` — submit for review. Returns redirect to payroll run detail with flash message.

### Events & Listeners

- PayrollRunSubmitted event fired after successful submission.
- Listener dispatches notification to all active finance users in the company.

### Mail & Notifications

- PayrollRunSubmitted notification via database channel: includes the payroll run name, period, and the actor who submitted it. Includes a link to the payroll run detail page.

### Tests

- Assert a draft run with employees submits successfully.
- Assert state changes to submitted after submission.
- Assert submitted by and submitted at are set correctly.
- Assert finance users receive a notification.
- Assert an empty run (no payslips) cannot be submitted (422).
- Assert a run with unresolved flags cannot be submitted (422).
- Assert a non-draft run cannot be submitted (422).
- Assert recomputation is triggered before submission.
- Assert only Owner and Payroll Officer can submit (Finance and Employee get 403).

## Acceptance Criteria

### Client-side

#### Behavior

- A "Submit for Review" button is visible on the payroll run detail page only when the run is in draft state
- The button is disabled if the run has zero employees, with a tooltip explaining why
- Clicking the button opens a confirmation dialog showing: Run Name, Period, Employee Count, Total Gross Pay, Total Deductions, and Total Net Pay
- A warning message explains that the run will be locked for editing after submission
- A "Confirm Submit" button submits the run, with a loading state during processing
- A "Cancel" button closes the dialog without making changes

#### Validation

- An error is shown if the run has no employees
- An error is shown if there are unresolved flags on any payslip
- An error is shown if the run is not in draft state

---

### Server-side

#### Side-effects

- Only draft runs can be submitted for review
- Submission is blocked if the run has no employees, with a clear error message
- Submission is blocked if there are unresolved flags on any payslip in the run
- A full payslip recomputation is triggered before the state transition
- The state transitions from draft to submitted on confirmation
- The submitted by field is set to the authenticated user and submitted at is set to the current timestamp
- Finance role users receive an in-app notification about the submitted payroll run
- Only Owner and Payroll Officer roles can submit a payroll run
- The payroll run becomes read-only after submission (bonuses, employee data, and absent days are locked)
- All changes are recorded in the audit trail
