# [GW] Payroll - Employee List Page with Filters and Pagination

## Description

Build the employee list page, the primary view for browsing and managing employees in the system. This includes a backend controller, a query class for building filtered/paginated queries, a form request for validating filter parameters, and an Inertia React page using the reusable data table component (PAY-023). The page displays key employee information with search and filter capabilities. Access is restricted to `owner` and `payroll_officer` roles.

## Technical Implementation

### What Models
- `app/Employee/Models/Employee.php` (existing model, no changes).
- Ensure scopes are defined: `scopeActive()`, `scopeSearch($term)`, `scopeFilterByDepartment($dept)`, `scopeFilterByStatus($status)`, `scopeFilterByPayBasis($basis)`.

### What Actions
- No domain actions in this ticket (read-only list page).

### Queries

#### `app/Employee/Queries/ListEmployeesQuery.php`
- A dedicated query class responsible for building the filtered, sorted, and paginated employee query.
- Constructor accepts validated filter parameters (from the Form Request).
- Methods:
  - `apply()`: returns the Eloquent Builder with all filters, sorting, and eager loads applied.
- Supported filters:
  - `search` (string): matches against `employee_number`, `first_name`, `last_name`, or `email` using `LIKE`.
  - `department` (string): exact match on `department` column.
  - `employment_status` (string): exact match on `employment_status` enum.
  - `pay_basis` (string): exact match on `pay_basis` enum.
- Supported sorting: any sortable column (`employee_number`, `first_name`, `last_name`, `department`, `position`, `employment_status`), default sort by `last_name` ascending.
- Eager loads: `compensation` (for displaying salary info), `compensation.currency`.
- Returns paginated results via `->paginate($perPage)`.

### Validation & DTOs

#### `app/Http/Requests/Employee/ListEmployeesRequest.php`
- Validates query string parameters:
  - `search`: `nullable|string|max:255`
  - `department`: `nullable|string|max:150`
  - `employment_status`: `nullable|string|in:active,inactive,on_leave,terminated`
  - `pay_basis`: `nullable|string|in:monthly,daily`
  - `sort`: `nullable|string|in:employee_number,first_name,last_name,department,position,employment_status`
  - `direction`: `nullable|string|in:asc,desc`
  - `per_page`: `nullable|integer|in:10,25,50,100`
  - `page`: `nullable|integer|min:1`

### Routes & Controllers

#### `app/Http/Controllers/Web/EmployeeController.php`
- `index()` method:
  - Authorize via middleware: `owner`, `payroll_officer` roles only.
  - Validate filters via `ListEmployeesRequest`.
  - Build query using `ListEmployeesQuery`.
  - Return Inertia response rendering `employees/index` with props:
    - `employees`: paginated employee data (shaped via API Resource).
    - `filters`: current active filter values (for pre-populating the filter bar).
    - `departments`: distinct department values for the filter dropdown.

#### Route definition (`routes/web.php`)
```php
Route::middleware(['auth', 'role:owner|payroll_officer'])->group(function () {
    Route::get('/employees', [EmployeeController::class, 'index'])->name('employees.index');
});
```

### API Resources

#### `app/Http/Resources/EmployeeListResource.php`
- Shape the employee data for the list page:
  - `id`, `employee_number`, `first_name`, `last_name`, `full_name` (computed accessor), `email`, `department`, `position`, `pay_basis`, `employment_status`, `employment_type`, `has_external_salary`, `date_hired`.
- Keep response lean; no nested compensation details in the list view.

### Policies & Authorization
- Route-level middleware restricts access to `owner` and `payroll_officer` roles.
- `employee` role cannot access this page and receives a 403 response.

### Frontend Components

#### `resources/js/pages/employees/index.tsx`
- Inertia page component using `authenticated-layout` as the layout.
- Page header: "Employees" title with employee count.
- Uses `DataTable` component (PAY-023) with columns:
  - **Employee #**: `employee_number` (sortable)
  - **Name**: `full_name` (`last_name, first_name`) (sortable)
  - **Department**: `department` (sortable)
  - **Position**: `position`
  - **Pay Basis**: `pay_basis` (badge: "Monthly" or "Daily")
  - **Status**: `employment_status` (colored badge: active=green, inactive=gray, on_leave=yellow, terminated=red)
  - **External Salary**: badge shown only if `has_external_salary` is true
  - **Actions**: link to employee detail/edit page
- Filter bar with:
  - Search input (debounced, searches name and employee number).
  - Department dropdown (populated from `departments` prop).
  - Status dropdown (active, inactive, on_leave, terminated).
  - Pay basis dropdown (monthly, daily).
- Pagination below the table.
- Empty state when no employees exist or no results match filters.
- "Add Employee" button in page header (links to create page, visible to owner/payroll_officer).

### Edge Cases & Notes
- Search should be case-insensitive.
- Department dropdown values should be dynamically derived from existing employee data, not hardcoded.
- The `full_name` accessor should format as "Last Name, First Name" for consistent sorting.
- Deactivated employees should still appear in the list (they are just filtered by status).
- Performance: ensure the query uses indexes on `company_id`, `employment_status`, and `department`.
