# [GW] Payroll - Set Up Domain-Driven Folder Structure

## Description

Establish the domain-driven directory structure for both the backend (Laravel) and frontend (React/Inertia). This creates all domain folders and their standard subdirectories so that subsequent tickets can place files in the correct locations. The autoloader must recognize the new namespaces.

## Technical Implementation

### Environment & Config

- Create 8 backend domain directories under the application root: Company, Employee, Payroll, Compensation, Currency, Bonus, CashAdvance, and Audit — each with standard subdirectories for Actions, Models, Policies, DTOs, Events, Listeners, Jobs, and Queries.
- Create HTTP layer directories for Controllers, Requests, and Resources.
- Create frontend directories for pages, component groups (UI, layout, forms, tables, feedback), layouts, hooks, types, and utilities.
- Update the autoloader configuration to include domain namespaces for improved IDE support.
- Place placeholder files in empty directories so version control tracks the structure.

### Tests

- Assert that all expected domain directories exist.
- Assert that classes in domain namespaces can be autoloaded correctly.

### Edge Cases & Notes

- The default User model should NOT be moved in this ticket — that happens in the authentication epic to avoid breaking default auth.
- No namespace collisions with the existing HTTP namespace.

## Acceptance Criteria

### Server-side

#### Side-effects

- All 8 backend domain directories exist with their standard subdirectories.
- All frontend directories exist for pages, components, layouts, hooks, types, and utilities.
- The autoloader recognizes domain namespaces without errors.
- No existing application functionality is broken (default routes, commands still work).
- The automated test suite passes without errors.
