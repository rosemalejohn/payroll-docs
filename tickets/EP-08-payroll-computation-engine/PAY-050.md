# [GW] Payroll - RecomputePayrollRunAction - Batch Recompute All Payslips

## Description

Implement a batch recomputation action that recalculates all payslips in a payroll run from scratch. This action deletes all existing line items, re-runs the compute payslip action for every employee in the run, and recalculates the run-level totals. It is only allowed on draft runs and must be idempotent (same inputs always produce the same outputs). This action is triggered whenever data changes that affect payslip computation.

## Technical Implementation

### What Actions

- Recompute Payroll Run action: validates the run is in draft state (rejects non-draft runs with an error). Deletes all existing line items for all payslips in the run using a bulk query. Resets all computed fields on each payslip to zero. Re-runs the compute payslip action for each payslip. Recalculates run-level totals (total gross, total deductions, total net, total bonuses, employee count) using aggregate SQL queries. The entire operation runs inside a database transaction for atomicity — if any single payslip computation fails, the entire operation rolls back.

### Tests

- Assert all payslips in the run are recomputed from scratch (line items deleted and regenerated).
- Assert run-level totals are updated after recomputation.
- Assert action only works on draft runs; non-draft runs are rejected.
- Assert line items are fully regenerated (old items deleted, new items created).
- Assert action is idempotent: running it twice with the same input data produces identical results.
- Assert the operation is wrapped in a database transaction for atomicity.
- Assert a failure in any single payslip computation rolls back the entire operation.
- Assert run totals use aggregate SQL queries for accuracy.
- Assert zero-employee runs complete without error, setting all totals to zero.

### Edge Cases & Notes

- For large runs (500+ employees), consider chunking the payslip iteration for performance.
- Line item deletion uses a bulk query rather than iterating per payslip.
- Run-level totals use aggregate SQL queries rather than PHP-side summation.
- This action is triggered by absent days changes, bonus changes, employee add/remove, compensation updates, and before run submission.

## Acceptance Criteria

### Server-side

#### Side-effects

- All payslips in the run are recomputed from scratch with line items deleted and regenerated
- Run-level totals (total gross, total deductions, total net, total bonuses, employee count) are updated after recomputation
- Only draft runs can be recomputed — non-draft runs are rejected with a clear error
- The action is idempotent — running it multiple times with the same data produces identical results
- The operation is wrapped in a database transaction — a failure in any payslip rolls back everything
- Run totals are calculated using aggregate SQL queries for accuracy
- The action is triggered automatically when data affecting computation changes (absent days, bonuses, employees, compensation)
