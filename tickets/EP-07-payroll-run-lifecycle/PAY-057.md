# [GW] Payroll - Submit Payroll Run for Review

## Description

Implement the action and UI for submitting a payroll run from `draft` to `submitted` state. Before submission, the system performs pre-submit validation (run has employees, all payslips computed, no unresolved flags) and triggers a full recomputation. A confirmation dialog presents a summary (employee count, total gross, total net) before the user confirms. After submission, the `submitted_by` and `submitted_at` fields are set, and finance role users receive a notification.

## Technical Implementation

### Action

**`app/Payroll/Actions/SubmitPayrollRunAction.php`**

- Accepts: `PayrollRun $payrollRun`, `User $actor`
- Returns: `PayrollRun` (updated)

**Pre-submit validation:**

1. Run must be in `draft` state; otherwise throw `InvalidStateTransitionException`.
2. Run must have at least one payslip (`payslips()->count() > 0`); otherwise throw `EmptyPayrollRunException`.
3. All payslips must be computed (no payslip with `gross_pay = 0` and `net_pay = 0` unless the employee has `has_external_salary = 1` with zero salary); otherwise throw `UncomputedPayslipsException`.
4. No unresolved flags exist on any payslip in this run (`PayslipFlag::where('payroll_run_id', ...)->where('status', 'open')->count() === 0`); otherwise throw `UnresolvedFlagsException`.

**Execution flow:**

```php
public function handle(PayrollRun $payrollRun, User $actor): PayrollRun
{
    // 1. Pre-submit validation
    $this->validate($payrollRun);

    // 2. Trigger full recomputation
    app(RecomputePayrollRunAction::class)->handle($payrollRun);

    // 3. State transition
    $payrollRun->state->transitionTo(SubmittedState::class);

    // 4. Set audit fields
    $payrollRun->update([
        'submitted_by' => $actor->id,
        'submitted_at' => now(),
    ]);

    // 5. Notify finance users
    $financeUsers = User::role('finance')
        ->where('company_id', $payrollRun->company_id)
        ->where('is_active', true)
        ->get();

    Notification::send($financeUsers, new PayrollRunSubmittedNotification($payrollRun));

    return $payrollRun->fresh();
}
```

### Controller

**`app/Http/Controllers/Web/PayrollRunController.php`** — add `submit` method:

- `submit(SubmitPayrollRunRequest $request, PayrollRun $payrollRun)` — Invokes `SubmitPayrollRunAction`. Returns redirect to payroll run detail with flash message.

### Route

```
POST /payroll-runs/{payrollRun}/submit -> PayrollRunController@submit
```

Middleware: `auth`, `role:owner|payroll_officer`.

### Form Request

**`app/Http/Requests/Payroll/SubmitPayrollRunRequest.php`**

- Authorization: user has role `owner` or `payroll_officer`.
- Additional authorization: payroll run must be in `draft` state.

### Notification

**`app/Payroll/Notifications/PayrollRunSubmittedNotification.php`**

- Channels: `database`, `mail` (optional based on config).
- Content: "Payroll run '{name}' for period '{period_name}' has been submitted for review by {actor_name}."
- Includes link to the payroll run detail page.

### Events

**`app/Payroll/Events/PayrollRunSubmitted.php`**

- Payload: `PayrollRun $payrollRun`, `User $actor`
- Fired after successful submission.
- Listener `SendPayrollRunSubmittedNotification` handles the notification dispatch.

### Frontend

**Confirmation Dialog:**

- Triggered when the user clicks "Submit for Review" button on the payroll run detail page.
- Dialog displays:
  - Payroll run name and period.
  - Employee count.
  - Total gross pay (formatted as currency).
  - Total net pay (formatted as currency).
  - Total deductions (formatted as currency).
  - Warning text: "This will lock the payroll run for editing. Bonuses and employee data can no longer be modified."
- "Confirm Submit" button sends POST request.
- "Cancel" button closes the dialog.
- Loading state on confirmation button during submission.

**Button visibility:**

- "Submit for Review" button only visible when `state === 'draft'`.
- Button disabled if run has zero employees (with tooltip: "Add employees before submitting").

### Error Handling

- `EmptyPayrollRunException` → 422 response with message "Cannot submit an empty payroll run. Add at least one employee."
- `UncomputedPayslipsException` → 422 with message "Some payslips have not been computed. Please wait for computation to complete."
- `UnresolvedFlagsException` → 422 with message "There are unresolved flags on this payroll run. Resolve all flags before submitting."
- `InvalidStateTransitionException` → 422 with message "This payroll run is not in draft state."

### Tests

- `tests/Feature/Payroll/SubmitPayrollRunTest.php`
  - Draft run with employees submits successfully.
  - State changes to `submitted`.
  - `submitted_by` and `submitted_at` are set correctly.
  - Finance users receive notification.
  - Empty run (no payslips) cannot be submitted (422).
  - Run with unresolved flags cannot be submitted (422).
  - Non-draft run cannot be submitted (422).
  - Recomputation is triggered before submission.
  - Only owner and payroll_officer can submit (finance/employee get 403).
  - Confirmation data (employee count, totals) is returned in pre-submit check.

## Acceptance Criteria

- [ ] Only payroll runs in `draft` state can be submitted.
- [ ] Submission is blocked if the run has no employees, with a clear error message.
- [ ] Submission is blocked if there are unresolved flags on any payslip in the run.
- [ ] A confirmation dialog displays the run summary: employee count, total gross, total net, and total deductions.
- [ ] A full payslip recomputation is triggered before the state transition.
- [ ] The state transitions from `draft` to `submitted` on confirmation.
- [ ] `submitted_by` is set to the authenticated user's ID and `submitted_at` is set to the current timestamp.
- [ ] Finance role users receive a notification about the submitted payroll run.
- [ ] Only owner and payroll_officer roles can submit a payroll run (403 for others).
- [ ] The payroll run becomes read-only after submission (bonuses, employee data, absent days locked).
