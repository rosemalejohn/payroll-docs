# [GW] Payroll - Payroll Run State Transitions

## Description

Implement the payroll run state machine using spatie/laravel-model-states. Payroll runs follow a linear workflow: draft -> submitted -> under_review -> finalized. Each transition is handled by a dedicated action that enforces authorization, validates preconditions, records metadata (actor and timestamp), and logs to the audit trail. State transitions are strictly one-directional and cannot be reversed (except via the separate "Return to Draft" feature for submitted/under review runs).

## Technical Implementation

### What Actions

- Submit Payroll Run action: validates the run is in draft state, validates the run has at least one employee, triggers payslip recomputation, sets submitted by/at fields, and transitions state from draft to submitted.
- Move to Under Review action: validates the run is in submitted state, authorization restricted to Finance or Owner roles, sets reviewed by/at fields, and transitions state from submitted to under review.
- Finalize Payroll Run action: validates the run is in under review state, authorization restricted to Owner role only, locks all payslips in the run, locks the associated payroll period, sets finalized by/at fields, and transitions state from under review to finalized.

### Routes & Controllers

- `POST /payroll-runs/{payrollRun}/submit` — submit for review.
- `POST /payroll-runs/{payrollRun}/review` — move to under review.
- `POST /payroll-runs/{payrollRun}/finalize` — finalize the run.

### Policies & Authorization

- Submit: Owner and Payroll Officer only.
- Move to Under Review: Finance and Owner only.
- Finalize: Owner only.
- All other roles receive 403 for each respective transition.

### Events & Listeners

- PayrollRunTransitioned event fired after each successful transition, captured by the audit trail, recording old state, new state, actor ID, actor name, actor role, and timestamp.

### Tests

- Assert states transition in the correct order only: draft to submitted to under review to finalized.
- Assert attempting an invalid transition (e.g., draft to finalized) is rejected.
- Assert attempting a backward transition (e.g., submitted to draft via the state machine) is rejected.
- Assert submission rejects runs with zero employees.
- Assert submission triggers payslip recomputation and sets submitted by/at.
- Assert moving to under review is restricted to Finance and Owner roles; other roles receive 403.
- Assert finalization is restricted to Owner role only; other roles receive 403.
- Assert finalization locks all payslips in the run.
- Assert finalization locks the associated payroll period.
- Assert audit trail records actor, old state, new state, and timestamp for every transition.
- Assert all transitions are idempotent-safe (re-submitting an already submitted run returns an error, not a crash).

## Acceptance Criteria

### Server-side

#### Side-effects

- Payroll runs transition through states in order only: draft to submitted to under review to finalized
- Invalid transitions (skipping states or going backwards) are rejected with a clear error
- Submitting a payroll run requires at least one employee — empty runs are blocked
- Submitting a payroll run triggers a full recomputation of all payslips before the state changes
- The submitted by, reviewed by, and finalized by fields record who performed each transition and when
- Finalization locks all payslips in the run, preventing further modifications
- Finalization locks the associated payroll period, preventing new runs from being created against it
- Only Owner and Payroll Officer can submit a run; only Finance and Owner can move to under review; only Owner can finalize
- All state transitions are recorded in the audit trail with the actor, old state, new state, and timestamp
