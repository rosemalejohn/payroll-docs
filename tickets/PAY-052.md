# PAY-052: AutoAttachBonusesToRunAction

## Description

Implement the action that automatically attaches applicable bonus types to a payroll run at creation time. When a new `PayrollRun` is created, the system iterates over all active bonus types (both system and custom) with `auto_trigger = 1` and attaches those whose `auto_trigger_month` matches the run's payroll period month (or is NULL, meaning every month). Each attached bonus creates a `payroll_run_bonuses` record with default values copied from the `bonus_types` catalog.

This action ensures that recurring bonuses like Meal Allowance (every month), 13th Month Pay (December), and June Special Bonus (June) are automatically included without manual intervention.

## Technical Implementation

### Action

**`app/Bonus/Actions/AutoAttachBonusesToRunAction.php`**

- Accepts: `PayrollRun $payrollRun`
- Returns: `Collection` of created `PayrollRunBonus` records

**Logic:**

```php
public function handle(PayrollRun $payrollRun): Collection
{
    $period = $payrollRun->payrollPeriod;
    $runMonth = $period->start_date->month;

    $eligibleBonuses = BonusType::query()
        ->where('company_id', $payrollRun->company_id)
        ->where('is_active', true)
        ->where('auto_trigger', true)
        ->where(function ($query) use ($runMonth) {
            $query->whereNull('auto_trigger_month')
                  ->orWhere('auto_trigger_month', $runMonth);
        })
        ->get();

    $attached = collect();

    foreach ($eligibleBonuses as $bonusType) {
        $runBonus = PayrollRunBonus::create([
            'payroll_run_id' => $payrollRun->id,
            'bonus_type_id'  => $bonusType->id,
            'bonus_code'     => $bonusType->code,
            'bonus_name'     => $bonusType->name,
            'amount'         => $bonusType->default_amount,
            'rate'           => $bonusType->default_rate,
            'percentage_basis'          => $bonusType->percentage_basis,
            'attendance_threshold_days' => $bonusType->attendance_threshold_days,
            'apply_to_all'   => true,
            'is_active'      => true,
            'added_by'       => auth()->id(),
        ]);

        $attached->push($runBonus);
    }

    return $attached;
}
```

### Integration Point

This action is called inside `CreatePayrollRunAction` (or equivalent run creation logic) after the `PayrollRun` record is persisted and currency snapshots are taken:

```php
// Inside CreatePayrollRunAction
$payrollRun = PayrollRun::create([...]);
app(SnapshotCurrencyRatesAction::class)->handle($payrollRun);
app(AutoAttachBonusesToRunAction::class)->handle($payrollRun);
```

### Models

**`app/Bonus/Models/PayrollRunBonus.php`**

- Fillable: `payroll_run_id`, `bonus_type_id`, `bonus_code`, `bonus_name`, `amount`, `rate`, `percentage_basis`, `attendance_threshold_days`, `apply_to_all`, `is_active`, `added_by`.
- Relationships: `belongsTo(PayrollRun::class)`, `belongsTo(BonusType::class)`, `hasMany(PayrollRunBonusExclusion::class)`.

### Edge Cases

- If a bonus type has been deactivated (`is_active = 0`) since the last run, it will not be auto-attached.
- If a `payroll_run_bonuses` record already exists for the same `payroll_run_id` and `bonus_type_id` (unique constraint), the action should skip that bonus type gracefully (use `firstOrCreate` or check before insert).
- Custom bonus types with `auto_trigger = 1` follow the exact same logic as system bonus types.
- The action does not trigger payslip computation — that is handled separately after employee assignment.

### Tests

- `tests/Feature/Bonus/AutoAttachBonusesToRunTest.php`
  - December run attaches Meal Allowance and 13th Month Pay, skips June Special.
  - June run attaches Meal Allowance and June Special, skips 13th Month.
  - March run attaches only Meal Allowance (no month-specific bonuses match).
  - Attendance Bonus is never auto-attached (auto_trigger = 0).
  - Inactive bonus types are not attached even if auto_trigger = 1.
  - Custom bonus with auto_trigger = 1 and auto_trigger_month = NULL is attached every month.
  - Custom bonus with auto_trigger = 1 and auto_trigger_month = 3 is attached only in March.
  - Default values (amount, rate, percentage_basis, attendance_threshold_days) are correctly copied from bonus_types.
  - `apply_to_all` defaults to 1 and `is_active` defaults to 1 on created records.
  - Duplicate attachment is prevented by unique constraint.

## Acceptance Criteria

- [ ] Meal Allowance (`auto_trigger=1`, `auto_trigger_month=NULL`) is attached to every payroll run regardless of month.
- [ ] 13th Month Pay (`auto_trigger=1`, `auto_trigger_month=12`) is attached only to payroll runs whose period starts in December.
- [ ] June Special Bonus (`auto_trigger=1`, `auto_trigger_month=6`) is attached only to payroll runs whose period starts in June.
- [ ] Attendance Bonus (`auto_trigger=0`) is never auto-attached; it must be manually added.
- [ ] Custom bonus types with `auto_trigger=1` follow the same month-matching logic.
- [ ] Default values (`default_amount`, `default_rate`, `percentage_basis`, `attendance_threshold_days`) are copied from `bonus_types` into the `payroll_run_bonuses` record.
- [ ] Created `payroll_run_bonuses` records have `apply_to_all=1` and `is_active=1` by default.
- [ ] Inactive bonus types (`is_active=0`) are not attached even if `auto_trigger=1`.
- [ ] The action handles duplicate attachment attempts gracefully without throwing errors.
- [ ] The action is called automatically during payroll run creation.
