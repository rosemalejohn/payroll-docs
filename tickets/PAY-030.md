# [GW] Payroll - Employee Deactivation with Confirmation

## Description

As an owner or payroll officer, I want to deactivate an employee with a confirmation step so that employees in open payroll runs are not accidentally removed, and deactivated employees are excluded from future runs while their historical data is preserved.

## Technical Implementation

### Action
- **`app/Employee/Actions/DeactivateEmployeeAction.php`**
  - Sets `employment_status = 'inactive'`
  - Before deactivating: queries for open payroll runs (draft/submitted/under_review) that include this employee
  - Returns list of affected runs if any exist (for confirmation UI)
  - If confirmed: deactivates the employee regardless of open runs

### Controller
- **`EmployeeController@checkDeactivation`** — `GET /employees/{employee}/check-deactivation` (returns affected runs)
- **`EmployeeController@deactivate`** — `PATCH /employees/{employee}/deactivate`

### Frontend
- "Deactivate" button on employee profile (owner and payroll_officer)
- Before deactivating: API call to check for open runs
- If open runs found: confirmation dialog listing affected runs with warning
- If no open runs: simple confirmation "Are you sure?"

### Business Rules
- Deactivated employees excluded from `CreatePayrollRunAction` auto-population
- Existing payslips in open runs are NOT removed (manual removal still required)
- Historical payslips of deactivated employees remain accessible
- Deactivation does not delete any data

### Tests
- Employee status changes to 'inactive'
- Confirmation required when employee is in open run
- Deactivated employee excluded from new run auto-population
- Historical payslips preserved after deactivation

## Acceptance Criteria

- [ ] Employee status set to `inactive` on deactivation
- [ ] Confirmation dialog warns about affected open payroll runs
- [ ] Deactivated employees excluded from future payroll run auto-population
- [ ] Historical payslips remain accessible and unchanged
- [ ] Employee data is NOT deleted (soft deactivation only)
- [ ] Owner and payroll_officer can deactivate; other roles cannot
