# [GW] Payroll - Cash Advance Recovery Post-Finalization

## Description

Implement the background job that automatically updates cash advance balances after a payroll run is finalized. The job processes all payslips with cash advance deductions, applies recovery amounts to the employee's active cash advances in FIFO order (oldest first), and automatically marks advances as fully recovered when their balance reaches zero.

## Technical Implementation

### Jobs

- Cash Advance Recovery job: dispatched by the finalization action after a payroll run is finalized. Queued on Redis.
- Processes each payslip with a cash advance deduction, finds the employee's active advances ordered by issued date (FIFO), and applies recovery amounts.
- Each employee's recovery is wrapped in a database transaction for consistency.

### Tests

- Assert a single advance is fully recovered (balance becomes zero, status changes to fully recovered).
- Assert partial recovery reduces the balance while keeping the status as active.
- Assert FIFO ordering: oldest advance is recovered first, then the next.
- Assert cascade across multiple advances when the deduction covers more than one.
- Assert employees with no active advances are handled without error.
- Assert payslips with zero cash advance deduction are skipped.
- Assert cancelled advances are not considered for recovery.
- Assert the job is idempotent (running twice does not double-recover).
- Assert balance never goes negative.
- Assert audit trail records all recovery amounts and status changes.

### Edge Cases & Notes

- FIFO recovery: advances are processed oldest-first by issued date, with creation date as tiebreaker.
- Partial recovery is supported — remaining balance carries forward to the next payroll cycle.
- Over-recovery is prevented: recovery amount per advance is capped at the advance's remaining balance.
- The job must be idempotent — verify recovery hasn't already been processed before applying.
- Only active advances are considered; cancelled advances are skipped.

## Acceptance Criteria

### Server-side

#### Side-effects

- After a payroll run is finalized, the system automatically updates cash advance balances for all employees who had cash advance deductions
- Recovery follows FIFO order — the oldest cash advance is recovered first
- When a cash advance's balance reaches zero, it is automatically marked as fully recovered
- Partial recovery is supported — if the deduction is less than the full balance, the remaining balance carries forward to the next payroll cycle
- If an employee has multiple active cash advances, recovery cascades through them in FIFO order until the deduction amount is fully applied
- Cancelled cash advances are skipped during recovery
- The recovery job runs automatically after finalization and does not need to be triggered manually
- Running the recovery job multiple times for the same payroll run does not result in double recovery
- All recovery changes (amount recovered, balance updates, status changes) are recorded in the audit trail
