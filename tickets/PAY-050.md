# [GW] Payroll - RecomputePayrollRunAction - Batch Recompute All Payslips

## Description

Implement a batch recomputation action that recalculates all payslips in a payroll run from scratch. We use two recomputation strategies to prevent performance bottlenecks:

1. **Full Recomputation (RecomputePayrollRunAction)** — Rebuilds every payslip in the run. Safe but slow. Used only for critical steps such as pre-submission.
2. **Granular Recomputation (RecomputeAffectedPayslipsAction)** — Recomputes only affected payslips and updates totals via adjustments. Default for most draft-run updates.

## Technical Implementation

### 1. Full Recomputation Strategy (The "Safety Net")

* Allowed only for draft runs.
* Deletes all existing line items for the run.
* Recomputes each payslip using the main computation action.
* Recalculates run totals using aggregation queries.

### 2. Granular Recomputation Strategy (The "Fast Path")

**RecomputeAffectedPayslipsAction**

* Accepts only the specific payslip IDs that changed.
* Recomputes those payslips.
* Computes the delta between previous and current totals.
* Passes these deltas to the totals updater.

**UpdatePayrollRunTotalsAction**

* Accepts adjustment deltas (e.g. changes in gross, deductions, net).
* Uses atomic increments/decrements to update totals.
* Avoids expensive full-run `SUM` operations.

## Triggers and Usage

* **Full Recompute**
  Triggered before submitting a payroll run for review.

* **Granular Recompute**
  Triggered when changes affect specific employees, such as:

  * Attendance changes
  * Bonuses
  * Currency rate updates
  * Salary adjustments

## Performance Considerations

* Granular recomputation significantly improves performance on large runs.
* Delta-based updates prevent repeated aggregate queries.
* Full recompute remains the final consistency checkpoint before submission.

## Acceptance Criteria

* [ ] Full recomputation rebuilds all payslips and totals.
* [ ] Granular recomputation updates only specified payslips.
* [ ] Totals adjust correctly using deltas.
* [ ] Isolated employee updates trigger granular recomputation.
* [ ] Run submission triggers a full recomputation.
* [ ] Both actions enforce draft-only restrictions.
* [ ] Performance improves for partial updates on large runs.
