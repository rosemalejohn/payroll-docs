# [GW] Payroll - Duplicate Payroll Run Prevention

## Description

Prevent the creation of duplicate payroll runs for the same pay period. When a user attempts to create a new run, the system validates that no existing run for the same period exists in draft, submitted, under review, or finalized state. If a duplicate is found, a clear validation error names the existing run and its current status. Cancelled runs do not block the creation of new runs. The payroll run creation form also displays recent runs for the selected period to proactively help users avoid duplicates.

## Technical Implementation

### What Actions

- Enhance the Create Payroll Run action with duplicate detection: before creating a run, query for existing runs matching the same period with overlapping dates where the status is not cancelled. If a match is found, reject creation with an error message naming the existing run and its status.

### Queries

- Find Overlapping Payroll Runs query: accepts period start and end dates, returns payroll runs where the period overlaps (not just exact match) and status is not cancelled. Overlap logic: existing start is before or on new end AND existing end is on or after new start.

### Validation & DTOs

- Enhance the Store Payroll Run request with a custom validation rule that checks for overlapping payroll runs.
- Custom No Overlapping Payroll Run validation rule using the overlapping runs query, with a failure message that includes the existing run name and status.

### Routes & Controllers

- Enhance the `GET /payroll-runs/create` route to include recent runs from the last 3 months in the page props.

### API Resources

- Recent Payroll Run resource: slim resource for the creation form, returning ID, name, period start, period end, status, and created date.

### Tests

- Assert creating a run with an overlapping period returns 422 when a draft run exists.
- Assert creating a run with an overlapping period returns 422 when a submitted run exists.
- Assert creating a run with an overlapping period returns 422 when an under review run exists.
- Assert creating a run with an overlapping period returns 422 when a finalized run exists.
- Assert creating a run for a period with only a cancelled run succeeds.
- Assert creating a run for a completely different period succeeds.
- Assert the validation error message includes the existing run name and status.
- Assert partial period overlap is detected (e.g., a run for March 1-15 blocks a run for March 10-20).
- Assert recent runs are returned in the create form page props.

### Edge Cases & Notes

- Period overlap detection must handle partial overlaps, not just exact date matches.
- Cancelled runs must be explicitly excluded from the overlap check.
- Race condition: two users creating runs for the same period simultaneously — consider a database-level unique constraint or advisory lock.
- The recent runs panel on the creation form should update dynamically as the user changes the period dates.

## Acceptance Criteria

### Client-side

#### Behavior

- "Recent Runs" panel shown after selecting period dates
- Overlap warning shows run name, status, and link
- Green checkmark if no existing runs for period
- Cancelled runs shown but don't block creation

#### Validation

- Error if non-cancelled overlapping run exists
- Partial period overlaps detected and blocked

---

### Server-side

#### Side-effects

- Overlapping active runs blocked with descriptive error
- Error includes existing run name and status
- Partial overlaps detected (e.g., Mar 1-15 blocks Mar 10-20)
- Cancelled runs don't block new run creation
- Recent 3-month runs provided on creation form
