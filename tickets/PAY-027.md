# PAY-027: Employee Create/Edit Page

## Description

Build the employee create and edit pages with full server-side validation, Inertia form handling, and redirect flows. This includes two controller methods for each operation (create/store and edit/update), dedicated form request classes, single-responsibility action classes for persistence, and Inertia React page components using the reusable form components (PAY-024). Successful mutations redirect to the employee list with a flash message.

## Technical Implementation

### What Actions

#### `app/Employee/Actions/CreateEmployeeAction.php`
- Single-responsibility action class using `lorisleiva/laravel-actions`.
- Accepts validated data (from Form Request).
- Creates a new `Employee` record with a generated ULID.
- Sets `company_id` from the authenticated user's company.
- Sets `hris_employee_id` to a generated ULID (for locally-created employees not yet synced from HRIS).
- Sets default payroll-specific fields: `pay_basis` from input, `has_external_salary = 0`, `has_mp2 = 0`, `has_sss_wisp = 0`, `hmo_amount = 0.00`.
- Returns the created `Employee` model.

#### `app/Employee/Actions/UpdateEmployeeAction.php`
- Accepts the `Employee` model and validated data.
- Updates only the allowed fields (HRIS-synced fields that can also be manually managed).
- Does not overwrite payroll-specific flags (those are managed in PAY-029).
- Returns the updated `Employee` model.

### What Models
- `app/Employee/Models/Employee.php` (existing).
- Ensure `$fillable` includes: `employee_number`, `first_name`, `last_name`, `middle_name`, `email`, `department`, `position`, `employment_type`, `employment_status`, `date_hired`, `pay_basis`.

### Validation & DTOs

#### `app/Http/Requests/Employee/StoreEmployeeRequest.php`
- Authorization: user must have `owner` or `payroll_officer` role.
- Rules:
  - `employee_number`: `required|string|max:50|unique:employees,employee_number`
  - `first_name`: `required|string|max:150`
  - `last_name`: `required|string|max:150`
  - `middle_name`: `nullable|string|max:150`
  - `email`: `nullable|email|max:255`
  - `department`: `nullable|string|max:150`
  - `position`: `nullable|string|max:150`
  - `employment_type`: `required|string|in:regular,contractual,probationary`
  - `employment_status`: `required|string|in:active,inactive,on_leave,terminated`
  - `date_hired`: `nullable|date|before_or_equal:today`
  - `pay_basis`: `required|string|in:monthly,daily`

#### `app/Http/Requests/Employee/UpdateEmployeeRequest.php`
- Authorization: user must have `owner` or `payroll_officer` role.
- Rules: same as `StoreEmployeeRequest` except:
  - `employee_number`: `required|string|max:50|unique:employees,employee_number,{employee_id}`
  - Uses `Rule::unique()->ignore($this->route('employee'))` for the current employee.

### Routes & Controllers

#### `app/Http/Controllers/Web/EmployeeController.php`
- `create()`:
  - Return Inertia response rendering `employees/create`.
  - Pass dropdown options: `employmentTypes`, `employmentStatuses`, `payBasisOptions`.
- `store(StoreEmployeeRequest $request)`:
  - Execute `CreateEmployeeAction` with validated data.
  - Redirect to `employees.index` with flash success message: "Employee created successfully."
- `edit(Employee $employee)`:
  - Return Inertia response rendering `employees/edit`.
  - Pass `employee` data and dropdown options.
- `update(UpdateEmployeeRequest $request, Employee $employee)`:
  - Execute `UpdateEmployeeAction` with employee and validated data.
  - Redirect to `employees.index` with flash success message: "Employee updated successfully."

#### Route definitions (`routes/web.php`)
```php
Route::middleware(['auth', 'role:owner|payroll_officer'])->group(function () {
    Route::get('/employees/create', [EmployeeController::class, 'create'])->name('employees.create');
    Route::post('/employees', [EmployeeController::class, 'store'])->name('employees.store');
    Route::get('/employees/{employee}/edit', [EmployeeController::class, 'edit'])->name('employees.edit');
    Route::put('/employees/{employee}', [EmployeeController::class, 'update'])->name('employees.update');
});
```

### Policies & Authorization
- Route-level middleware restricts to `owner` and `payroll_officer`.
- Form Request `authorize()` method provides a second authorization check.

### Frontend Components

#### `resources/js/pages/employees/create.tsx`
- Inertia page component using `authenticated-layout`.
- Page header: "Add Employee" with breadcrumb back to employee list.
- Uses Inertia `useForm` with initial empty values for all fields.
- Form sections (using `FormSection` from PAY-024):
  - **Personal Information**: `employee_number`, `first_name`, `last_name`, `middle_name`, `email`.
  - **Employment Details**: `department`, `position`, `employment_type` (select), `employment_status` (select), `date_hired` (date picker), `pay_basis` (select).
- Form actions: "Save" button (submit), "Cancel" button (links back to employee list).
- On submit: `form.post(route('employees.store'))`.
- Validation errors from Laravel display inline under each field.
- Submit button disabled while `processing` is true.

#### `resources/js/pages/employees/edit.tsx`
- Same layout as create page but pre-filled with existing employee data.
- Page header: "Edit Employee - {employee.full_name}" with breadcrumb.
- Uses Inertia `useForm` initialized with the employee's current values.
- On submit: `form.put(route('employees.update', employee.id))`.
- All other behavior identical to create page.

#### Shared form component (optional refactor)
- If the create and edit forms are identical, extract a shared `EmployeeForm` component that accepts `initialData`, `submitUrl`, `submitMethod`, and `pageTitle` as props.

### Edge Cases & Notes
- `employee_number` uniqueness is scoped to the `employees` table globally (not per company in this implementation, since single-company deployment).
- The create page must not show payroll-specific flags (`has_external_salary`, `has_mp2`, etc.) as those are managed in PAY-029.
- `hris_employee_id` is auto-generated for manually created employees and should not be editable in the UI.
- Middle name is optional and should be handled gracefully (null in the database, empty string in the form).
- Route model binding uses ULID for the `{employee}` parameter.
