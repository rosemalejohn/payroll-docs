# [GW] Payroll - Payroll Period Management

## Description

Build the payroll period management feature including the payroll periods database table and model. Payroll periods define the date ranges and working day counts for payroll processing cycles (e.g., "March 2026"). Each period belongs to a company and can be monthly, semi-monthly, or weekly. The Owner and Payroll Officer can create and manage periods. A locked period prevents modifications and new payroll runs from being created against it, ensuring finalized periods remain immutable.

## Technical Implementation

### What Migrations

- Create the payroll periods table with ULID primary key, company reference, name, period type (monthly/semi_monthly/weekly), start date, end date, working days (default 22), locked flag (default false), creator reference, and standard timestamps.
- Index on company reference for query performance.

### What Models

- Create PayrollPeriod model in the Payroll domain with ULID support, factory, and audit trail integration.
- Relationships: belongs to Company, has many PayrollRuns, belongs to creator (User).
- Scopes: unlocked periods, locked periods, filter by company, filter by period type.
- Helpers: check if period is locked, check if a new run can be created, lock the period.
- Factory with states for monthly, semi-monthly, weekly, and locked.

### What Actions

- Create Payroll Period action: creates a new period with the given date range and working days.
- Update Payroll Period action: updates a period only if it is not locked.

### Validation & DTOs

- Store Payroll Period request: validates name (required), period type (required, must be monthly/semi_monthly/weekly), start date (required), end date (required, must be after start date), and working days (required, minimum 1, maximum 31).
- DTO for payroll period data using spatie/laravel-data.

### Routes & Controllers

- `GET /payroll-periods` — paginated list with name, type, date range, working days, and locked status.
- `POST /payroll-periods` — create new period.
- `GET /payroll-periods/create` — create form.
- `GET /payroll-periods/{period}/edit` — edit form (disabled fields if locked).
- `PUT /payroll-periods/{period}` — update period.

### Policies & Authorization

- Owner and Payroll Officer can create and manage payroll periods.
- Locked periods cannot be edited.

### Tests

- Assert a payroll period can be created with valid data and correct initial state.
- Assert validation prevents start date from being on or after end date.
- Assert locked periods cannot be edited.
- Assert working days defaults to 22.
- Assert period type only accepts monthly, semi-monthly, or weekly.
- Assert the unlocked and locked scopes filter correctly.
- Assert the period list is paginated and shows locked status.
- Assert all changes are captured in the audit trail.

### Edge Cases & Notes

- Working days defaults to 22 (standard Philippine monthly working days) but can be adjusted for periods with holidays or partial months.
- Period date ranges are not validated for overlap at the database level; this is enforced at the application/validation layer.
- The locked flag is the primary mechanism to prevent new payroll runs from being created against a period — actions should check this before allowing run creation.

## Acceptance Criteria

### Client-side

#### Layout

- Accessible from sidebar for Owner and Payroll Officer

#### Behavior

- List shows name, type, dates, working days, lock status
- Create form: Name, Type, Start/End Date, Working Days
- Locked periods show lock icon, edit disabled
- Unlocked periods can be edited

#### Validation

- Error if start date is on or after end date
- Error if period type is invalid
- Error if working days outside 1-31 range

---

### Server-side

#### Side-effects

- New periods default to unlocked, 22 working days
- Locked periods reject edit attempts
- Only Owner and Payroll Officer can manage periods
- All changes recorded in audit trail
