# [GW] Payroll - In-App Notification System

## Description

Implement an in-app notification system using Laravel's built-in notification framework with the database channel. This includes creating the notifications table, a notification controller for listing and marking notifications as read, and a frontend notification bell icon in the application header with an unread count badge and dropdown panel. The unread count is delivered via Inertia shared props so it is available on every page load.

## Technical Implementation

### What Migrations

- Notifications table (Laravel default): UUID primary key, type, notifiable morph columns, data (JSON payload), read_at (nullable timestamp), timestamps. Indexes on notifiable type/ID and read_at for unread count queries.

### What Models

- User model: ensure Notifiable trait is present, add helper method for unread notification count.

### What Actions

- Mark Notification as Read action: marks a single notification as read for the authenticated user.
- Mark All Notifications as Read action: marks all unread notifications as read for the authenticated user.

### Validation & DTOs

- Mark Notification Read request: validates the notification belongs to the authenticated user.

### Routes & Controllers

- `GET /notifications` — paginated notification list for the authenticated user, ordered newest first.
- `PATCH /notifications/{notification}/read` — mark a single notification as read.
- `POST /notifications/mark-all-read` — mark all unread notifications as read.

### API Resources

- Notification resource: includes ID, human-readable type label, data payload (title, message, action URL), read status, and formatted timestamp with relative time.

### Policies & Authorization

- All notification routes require authentication.
- Users can only view and interact with their own notifications.

### Tests

- Assert authenticated user can view their notifications.
- Assert authenticated user can mark a notification as read.
- Assert authenticated user can mark all notifications as read.
- Assert user cannot mark another user's notification as read.
- Assert unauthenticated user is redirected to login.
- Assert unread count is included in Inertia shared props.
- Assert recent notifications are included in Inertia shared props.
- Assert notifications are sorted by newest first.
- Assert pagination works correctly on the full notifications page.

### Edge Cases & Notes

- The unread count query runs on every Inertia request via shared props — ensure it is efficient with indexed read_at column and count query only.
- Recent notifications in shared props should be limited to 5 items to keep payload small.
- Notification data JSON must follow a consistent structure (title, message, action_url) across all notification types.

## Acceptance Criteria

### Client-side

#### Layout

- A notification bell icon appears in the application header for all authenticated users

#### Behavior

- The bell icon displays an unread count badge when there are unread notifications
- The badge is hidden when the unread count is zero
- Clicking the bell opens a dropdown panel showing the most recent notifications
- Each notification shows a title, message preview, and relative timestamp
- Unread notifications are visually distinct from read notifications
- Clicking a notification marks it as read and navigates to the relevant page
- A "Mark all as read" option is available in both the dropdown and the full notifications page
- A "View all notifications" link in the dropdown navigates to the full notifications page
- The full notifications page displays all notifications with pagination
- An empty state message is shown when there are no notifications

---

### Server-side

#### Side-effects

- The notifications table is created and stores all in-app notifications
- Unread notification count is available on every page load via shared props
- Users can only see and interact with their own notifications
- Marking a notification as read updates its read timestamp
- Marking all as read updates all unread notifications for the user
- Notifications are sorted by newest first
